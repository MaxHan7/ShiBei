# 拾贝 V2 核心技术框架通俗解释

这份文档不用工程术语来解释：我们为什么要把拾贝 V2 的 AI 出题系统设计成现在这个样子。

如果一句话概括：

> 拾贝 V2 不应该让 AI “凭感觉写几道题”，而应该让 AI 像一条清晰的生产线一样，先理解文章，再拆知识点，再判断要观察用户是否理解什么，最后才写题。

## 1. 我们已经有了 ECD，它解决什么问题？

ECD 是我们的“出题设计原则”。

它告诉我们：

- 不是先写题。
- 而是先问：我们希望用户学会什么？
- 再问：用户做出什么表现，能证明他真的理解了？
- 最后才问：用什么题型最适合让用户表现出这种理解？

比如 DMC 模型：

错误做法：

> 看到 DMC，就随便问“DMC 是什么？”

ECD 做法：

> 先判断用户真正需要理解的是：DMC 三层分别承担什么设计作用。  
> 然后判断：如果用户能把三层和对应作用匹配起来，就说明他理解了这个结构。  
> 所以这里适合出连线题。

ECD 解决的是“为什么这样出题”。

## 2. 但 ECD 还不够，它不解决工程问题

ECD 不会自动告诉我们：

- prompt 怎么拆？
- 每一步传多少原文？
- JSON 怎么稳定输出？
- 失败了怎么重试？
- 怎么判断一次改动让质量变好了还是变差了？
- 怎么避免 prompt 越写越长？

这些是技术框架要解决的问题。

## 3. 技术侧我们对标的是 DSPy-style LM Program

你可以把 DSPy-style 理解成一句话：

> 不要把 AI 系统写成一大段 prompt，而要把它写成一套可以测试、可以评估、可以优化的程序。

它的思想很适合我们。

因为拾贝现在的问题不是“缺一句神奇 prompt”，而是：

- 有些步骤职责混在一起。
- 有些步骤吃了太多上下文。
- 有些步骤输出太多中间字段。
- 改了 prompt 之后，不知道到底哪里变好了、哪里变差了。

DSPy-style 给我们的启发是：

```text
每一步都像一个小函数。
每个小函数有明确输入。
每个小函数有明确输出。
每个小函数都有评估指标。
整个系统像程序一样运行，而不是像一段长 prompt 一样许愿。
```

## 4. 我们的系统为什么不能做成开放式 Agent？

开放式 agent 像这样：

> AI 自己决定下一步做什么，自己调用工具，自己规划路径。

这不适合拾贝当前阶段。

因为我们需要的是稳定产出：

```text
文章 -> 知识点 -> 小知识点 -> 题目计划 -> 题目
```

这个流程不应该每次都让 AI 自己重新决定。

所以拾贝更适合：

> 固定 workflow + 每一步小模型调用。

也就是说，流程由我们控制，AI 只在每个清楚的小任务里发挥能力。

## 5. 拾贝 V2 的“生产线”应该长什么样？

可以把它想象成一条出题生产线。

### 第一步：切原文

AI 或程序先把文章切成稳定段落。

输出：

```text
p-001
p-002
p-003
...
```

这一步不总结、不出题。

### 第二步：切大知识点

根据整篇文章，判断哪些内容应该成为用户复习路径上的知识点。

输出：

```text
游戏化的概念
DMC 模型
阶段性目标
挑战与能力匹配
...
```

这一步只决定大知识点，不拆每个知识点内部的小点。

### 第三步：拆小知识点

对每个大知识点，再拆里面有哪些小的可学习内容。

比如 DMC 模型里可能有：

```text
DMC 是分层框架
动力层负责体验方向
机制层负责参与规则
组件层负责可见界面元素
不要把游戏化理解成堆积分徽章
```

这一步不出题，只做知识 inventory。

### 第四步：做 ECD 规划

这一步问：

> 哪些小知识点值得考？  
> 用户要表现出什么，才说明他理解了？  
> 用选择题还是连线题更合适？

输出不是题目正文，而是题目计划。

比如：

```text
任务 1：用选择题检查用户是否理解 DMC 不是组件清单。
任务 2：用连线题检查用户能否匹配三层和对应作用。
```

### 第五步：生成题目

现在才开始真正写用户能看到的题。

选择题模块只写选择题。  
连线题模块只写连线题。  
它们不能擅自改变上一步的题目计划。

### 第六步：生成单元总结

题目都生成后，再写这个知识点的开场和总结。

### 第七步：生成报告

报告不是给用户看的，是给我们检查系统质量看的。

它告诉我们：

- 哪个阶段慢？
- 哪个阶段 JSON 失败？
- 哪个知识点漏了？
- 哪个题型选择不合理？
- 哪道题的 source anchor 不准？

## 6. 为什么这套框架能解决我们现在的问题？

### 问题一：JSON 不稳定

以前：

> 一个 prompt 要输出太多东西，容易截断或格式乱。

现在：

> 每一步只输出小 JSON，结构更稳定。

### 问题二：性能慢

以前：

> 每个阶段都反复读整篇文章。

现在：

> 只有切知识点时看整篇；具体到某个知识点时，只看对应原文片段。

### 问题三：质量忽高忽低

以前：

> 模型可能直接跳到写题，漏掉中间的知识建模。

现在：

> 先拆知识点，再拆小知识点，再做 ECD 题目计划，再写题。

### 问题四：prompt 越写越长

以前：

> 发现一个问题，就往 prompt 里加一句规则。

现在：

> 如果一个 prompt 变得太复杂，就拆成新的模块，或者把无用输出删掉。

### 问题五：不知道改动有没有变好

以前：

> 看一眼报告，凭感觉判断。

现在目标：

> 固定黄金文章、固定指标、固定对比方式。

## 7. 为什么现在不直接用 DSPy 框架？

因为 DSPy 主要是 Python 生态，而我们的 V2 后端现在是 Node。

更重要的是，我们还在确定字段、流程、质量标准。如果现在直接迁移框架，会增加很多变量。

所以当前做法是：

> 不直接使用 DSPy 运行时，但采用 DSPy 的思想来重构我们的 Node 后端。

也就是：

- 用“模块”思考 prompt。
- 用“输入输出合同”定义每一步。
- 用“指标”评估每一步。
- 用“黄金样本”测试每次改动。

## 8. BAML、LangGraph、Structured Outputs 分别是什么位置？

它们不是主框架，但很有价值。

### BAML

它像是 prompt 和 schema 的工程管理工具。

我们借鉴它：

- prompt 要像代码一样管理。
- schema 要和 prompt 绑定。
- 每个 prompt 要有测试。

### LangGraph

它更像长流程调度框架。

我们以后可能借鉴它：

- 任务状态持久化。
- 某个阶段失败后恢复。
- 长任务可观察。

但它不是当前最核心的出题质量问题。

### Structured Outputs

它解决的是结构化输出稳定性。

我们借鉴它：

- 每个阶段必须有 schema。
- 输出不能只靠 prompt 口头要求。
- 解析失败应该走工程 retry，而不是靠用户等待。

## 9. 下一步应该怎么做？

最现实的下一步不是继续讨论框架，而是把这套框架落到三个具体文件/标准里。

### 1. Stage Contract 表

写清楚每一步：

- 输入什么。
- 输出什么。
- 不能输入什么。
- 不能输出什么。
- 看多少原文。
- 失败怎么处理。

### 2. Context Passing 规则

解决 token 浪费：

- 哪一步能看整篇文章。
- 哪一步只能看当前知识点片段。
- 当前片段不够时怎么扩展。

### 3. Eval 标准

把“看起来质量好”变成更稳定的判断：

- DMC 是否独立。
- DMC 是否有连线题。
- nodeLabel 是否短。
- JSON 是否成功。
- 每阶段耗时是多少。
- 题目数量有没有塌缩。

## 10. 最核心的一句话

我们现在要做的不是“写一个更长、更聪明的 prompt”。

我们要做的是：

> 把拾贝的 AI 出题系统变成一套有教育设计原则、有工程模块、有输入输出合同、有评估指标的 LLM program。

ECD 负责教育正确性。  
DSPy-style LM Program 负责工程稳定性。  

这两套东西合在一起，才是拾贝 V2 后端生成系统的核心技术框架。
