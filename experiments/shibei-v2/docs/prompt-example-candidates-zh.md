# V2 Prompt Example Candidates

这个文档先保存我们在 V2 样章讨论中发现的正反例。它不是当前 prompt 的一部分，也不是硬性生成规则。后续正式设计 V2 prompt 时，以 `v2-prompt-field-rules-zh.md`、`golden-samples/README.md` 和 V2 golden samples 为准；如果某一类问题仍不稳定，再从这里挑少量代表例子加入对应 prompt。

## 使用原则

- 这些例子只作为备选 few-shot / anti-shot 材料。
- 不建议一次性全部塞进 prompt，避免 prompt 变重或让模型过拟合样例。
- 若使用反例，必须明确标注“不要这样写”，并配套改写后的正例。
- 每个例子最好带上问题类型标签，让模型学到判断标准，而不是只模仿表面句式。

## 例子 1：轻量题不能依赖隐含上下文

问题原句：

> 这里的“产品品味”首先落到哪个动作上？

问题：

“这里的”依赖用户记得上一页或原文段落的上下文。用户单独看到题目时，不一定知道它指的是哪一段、哪个语境。

改写方向：

> 让 agent 判断用户痛点前，PM 最先要做什么？

为什么更好：

它保留了轻量感，但给出了最小动作场景。用户不需要回忆原文位置，也能知道题目在问什么。

## 例子 2：匹配题要把抽象词换成可理解对象

问题原句：

> 把反馈来源和它能补充的上下文匹配起来。

问题：

“上下文”太抽象，用户可能不知道是在指什么上下文。题干虽然短，但理解入口不够清楚。

改写方向：

> 把这些反馈来源和它们提供的线索匹配起来。

为什么更好：

“线索”比“上下文”更贴近用户感知；右侧选项再写成“明确提出的问题 / 社区里的抱怨和讨论 / 产品内发生的行为线索”，就能保持题干轻量，同时补足作答语境。

## 例子 3：避免让用户回忆“这篇文章怎么说”

问题原句：

> 你让 agent 根据 GitHub issue 生成优先级报告。哪种提示更接近这篇文章里的 PM 工作方式？

问题：

“这篇文章里的”会把题目变成阅读理解式回忆，默认用户还记得文章原文的表述。

改写后：

> 你让 agent 根据 GitHub issue 生成优先级报告。哪种提示更能让输出可检查？

为什么更好：

题目直接考知识点本身：PM 要把任务讲到机器可执行、可输出、可检查，而不是考用户是否记得文章怎样表达。

## 例子 4：抽象判断题也要摆脱“文章视角”

问题原句：

> 这篇文章里，eval 更像什么？

问题：

题干仍然在引导用户回忆文章，而不是直接进入知识点。

改写后：

> 在 AI 产品质量管理里，eval 更像什么？

为什么更好：

它把问题放回知识场景，让用户判断 eval 的角色：反馈回路，而不是一次性验收表。

## 例子 5：场景题不要写成“符合文章思路”

问题原句：

> 客户提出一个新需求，团队希望当天判断能否推进。哪种做法更符合这篇文章的思路？

问题：

“符合这篇文章的思路”仍然要求用户回忆文章观点，题目本身没有把考点说清楚。

改写后：

> 客户提出一个新需求，团队希望当天判断能否推进。哪种做法更能形成可验证的判断？

为什么更好：

它直接考察“反馈、agent、trace、eval 闭环让判断可验证”的核心理解。

## 例子 6：行动建议题要说清楚要实践的对象

问题原句：

> 你只有一个下午，想实践这篇文章的方法。最合理的第一版范围是什么？

问题：

“这篇文章的方法”太泛，需要用户回忆整篇文章的方法是什么。

改写后：

> 你只有一个下午，想跑通一条最小 AI PM 反馈闭环。最合理的第一版范围是什么？

为什么更好：

题干用“最小 AI PM 反馈闭环”直接点出要实践的对象，用户可以围绕反馈源、agent、trace、eval 做判断。

## 例子 7：轻量题优先正向提问

问题原句：

> PM 和工程师边界变小，并不意味着什么？

问题：

“并不意味着什么”是负向提问。用户要先理解概念，再做语义反转，会增加不必要的认知负担。

改写后：

> PM 和工程边界变近后，更重要的变化是什么？

为什么更好：

题目让用户直接找正确理解。错误选项仍然可以承载误区，但题干不把用户绕进去。

## 例子 8：行动题不必用“最不应该”

问题原句：

> 你准备让 agent 帮团队找下个版本的最大痛点。最不应该只喂给它哪一种材料？

问题：

这是负向提问，而且考点不是边界排除，没必要让用户先处理“最不应该”。

改写后：

> 你准备让 agent 帮团队找下个版本的最大痛点。哪种材料最能支撑它做出可回溯判断？

为什么更好：

它把用户注意力放到正确行动上：提供用户原话、工单链接和产品行为数据，让判断可回溯。

## 例子 9：两小时实操题不必问“关键不是”

问题原句：

> 两小时实操的关键不是先做什么？

问题：

这同样是负向题干，而且不是为了排除一个复杂边界，只是在问第一步行动。

改写后：

> 只有一个下午想跑通 AI PM 小闭环，第一步更适合做什么？

为什么更好：

它直接引导用户选择起步动作：选一个真实反馈源，而不是等待完整平台。

## 例子 10：知识点开场和第一题不能同义复述

问题组合：

> 知识点：AI 时代的 PM 不一定每天写业务代码，但要能把产品、数据源、输出格式和判断标准讲清楚，让 agent 真正执行任务。

> 轻量题：一个 PM agent 任务最需要被讲清楚的是什么？

问题：

知识点开场和第一题几乎承担同一个表达功能。用户会感觉刚读完答案，又被问了一遍。

改写方向：

> 你让 agent 根据 GitHub issue 生成优先级报告。哪种提示更能让输出可检查？

为什么更好：

知识点开场负责概括核心理念；第一题负责让用户主动切入。两者可以围绕同一个核心，但题目应换成判断、选择、匹配或场景化提问，而不是把开场句改写成问句。

## 例子 11：Summary 不要写成目录

问题写法：

> 文章介绍了 Hook 的定义、使用场景，以及它和 Prompt、CI 的区别。

问题：

这像目录，不像记忆唤醒锚点。它告诉用户文章列了哪些点，但没有点出文章的核心命题。

改写后：

> AI 已经能很快把想法变成 Demo，但真正难的是让它每次都稳定、可控、可复用。Hook 的作用，是把那些不能只靠人提醒的检查、限制和自动动作，固定到 AI 编程流程里。它和 Prompt、规则文档、CI 各自承担不同职责，帮助 PM 从“会让 AI 写代码”走向“知道怎样让 AI 可靠地工作”。

为什么更好：

先点核心命题，再轻量带出展开方向。用户更容易立刻想起这篇文章在解决什么问题。

## 例子 12：连线题不要只连名词和空泛解释

问题写法：

> 把需要 Hook 的信号含义匹配起来。

问题：

如果右侧只是信号解释，用户容易停留在机械对齐。文章真正要考的是：每个信号出现时，Hook 可以起到什么作用。

改写方向：

> 把需要 Hook 的信号和对应作用匹配：

- AI 开始碰危险动作 → 执行前检查风险，必要时阻止或升级人工确认
- 同样的提醒已经说了很多遍 → 把反复提醒变成项目级规则或自动动作
- Demo 要交给别人继续改 → 把 lint、format、测试等协作秩序固定到仓库流程
- AI session 变长 → 结束前验收测试、后台任务、未提交变更等状态

为什么更好：

它考的是关系理解和具体化理解，而不是让用户背几个信号名称。

## 后续使用建议

后续如果要把这些例子放进 prompt，建议按生成环节分组引入：

1. Summary 生成只放 summary 正反例。
2. 轻量理解题生成只放题干自足、正向提问、开场不复述的例子。
3. 匹配题生成只放关系匹配和手机端简洁表达的例子。
4. 干扰项生成优先遵守 `v2-prompt-field-rules-zh.md` 的生成顺序与干扰项规则；只有效果不稳定时，再补充这里的误区与干扰项相关案例。
