"在高级工程师基准上，GPT-5.5 得了 62.5 分。"

"用 Opus 4.7 写计划，GPT-5.5 就会变成一头猛兽。"

"它有足够的胆量，真的会删掉一堆文件。"

Every 在 GPT-5.5 发布前做了三周内部测试，测试范围覆盖编码、写作、知识工作和 OpenClaw。主讲人把这期称为一次 “vibe check”：不只看跑分，也看它在真实团队里能不能完成 Dayline、Monologue、Cora 这类产品任务。最扎眼的结论来自编码：GPT-5.5 已经能明显拉开和 Opus 4.7 的差距，但它最好用的方式并非单独上阵，而是先拿到一份足够具体的计划。

Every 的测试场景很接近真实团队：他们自己用 Codex 和 Claude Code 做 Cora、Sparkle、Monologue 等内部应用，也给公司管理层做 AI 培训。换成中文互联网语境，这期不只是模型发布评测，更像一份工程团队怎么分配模型岗位的实战笔记。它把模型能力放回了排期、交付、重构和写作这些具体工作里，也放回每天开工时的工具选择现场中。

62.5 分把模型推进高级工程现场

Every 做了一个 Senior Engineer Benchmark，简称 SE Bench。它把一个真实应用 Proof 的 “vibe-coded slop” 代码库交给模型，要求模型从第一性原理重写。评分标准来自两名真实高级工程师的改写结果。人类高级工程师通常能拿到 80 到 90 分，GPT-5.5 最高跑到 62.5 分，Opus 4.7 则稳定在 30 分出头。

"人类高级工程师在这个基准上通常能得 80 或 90 分。我们见过的最好成绩是 GPT-5.5，它得了 62.5 分。"

这个分数还没到人类高级工程师区间，但主讲人强调，SE Bench 还没有被刷爆，前沿仍然有很大空间。GPT-5.5 单独写计划时大概能到低到中 40 分；多做提示后能接近低到中 50 分。真正的跃升发生在另一种组合里：让 Opus 4.7 先写计划，再让 GPT-5.5 执行。

主讲人特意说明，这不是一个刷 LeetCode 或修小 bug 的测试。Proof 是他自己做过的应用，代码里带着 AI 快速生成后的混乱痕迹。模型要交付一套能和人类工程师改写结果比较的完整实现。62.5 分的意义在于，它第一次让模型接近了“能重写系统”的位置。

先让 Opus 写计划，再让 5.5 动手

主讲人提到一个搭配：Opus 4.7 负责规划，GPT-5.5 负责执行。原因很直接，Opus 4.7 写出的计划更像工程合同，会说明哪些概念必须成立、哪些文件要删、重写完成后某个大文件大约应该剩多少行。GPT-5.5 拿到这样的计划后，不会只补几个洞，而会沿着计划把改造做完。

"用 Opus 4.7 写计划，GPT-5.5 就会变成一头猛兽。"

Opus 4.7 自己执行这份计划时，反而会退回到小修小补。主讲人说，它会觉得任务太费力，然后只挑一小块处理。GPT-5.4 也有类似倾向，只是做得比 Opus 稍好。GPT-5.5 的不同在于，它在 extra high reasoning 下更敢动手，能把数小时级别的任务从头推进到尾。

这种分工给工程经理一个很具体的操作方式：需求进入 AI 流程后，先用擅长结构化规划的模型写改造合同，再把合同交给更愿意长时间执行的模型。原来团队会争论“Claude 强还是 GPT 强”，Every 这次测试更像把争论拆开：谁写计划，谁动手，谁最后验收。

SE Bench 看的是重写烂代码的胆量

SE Bench 的重点不在补 bug，而在判断模型有没有高级工程师的结构感。模型拿到的是一个真实但混乱的应用代码库，任务是判断底层不变量、重新划分边界、让代码回到概念清晰的状态。很多模型会被现有文件牵着走，在局部打补丁；GPT-5.5 更愿意从核心原则出发，改掉结构。

"它有那种果断、胆量和自主性，真的会删掉一堆文件，然后从头开始。"

这对工程团队的实际意义很明确：当代码库已经被 AI 写坏，继续让模型局部修补只会堆出更多噪声。GPT-5.5 的优势在于重构胆量。它仍然离高级工程师有大约 30 分差距，但第一次让 Every 团队觉得，模型在接近“理解结构再重写”的工作方式。

主讲人用了一个很工程化的词：invariants，不变量。好的重写会先找出代码库里必须保持成立的规则，再围绕规则重建结构；旧目录和报错信息只能作为线索，不能变成施工图。GPT-5.5 让他满意的地方，正是它不会被旧代码牵着走太远。

好计划要写到文件数量和删除范围

GPT-5.5 自己写的计划更适合人读，概念基本都有，但篇幅偏长，缺少程序员式的契约细节。Opus 4.7 的计划则更短、更硬，像一份清单：做成后某个文件不该超过 100 行，哪些旧文件该删除，什么状态才算“足够好”。主讲人认为，这种精确度正好能激发 GPT-5.5 的执行力。

"Opus 的计划会告诉模型：如果重写足够好，这个大文件只会剩 100 行。"

所以，5.5 的使用姿势也随之变化。把它当成聊天模型，收获可能只是一个不错的回答；把它放进有约束的工程流程，输出会更像能交付的工作。计划越具体，GPT-5.5 越敢执行；计划越像“请你发挥”，它越容易浪费在不必要的探索上。

有意思的是，GPT-5.5 的计划更像写给人看的文档，概念齐全、语言顺滑，却少了“完成定义”。Opus 4.7 的计划更干、更像机器人，但这种机器味在工程执行里反而有用。模型协作的经验变成了：别只追求提示词优雅，要把验收条件写到足够硬。

Dayline 和 Monologue 跑出了真实项目速度

Every 的 Naveen 用 GPT-5.5 做了两个实测。一个是 Dayline，一款原生 iOS 和 Mac 待办应用。他的反馈是，5.5 能按计划持续推进，把一串功能做到完成，而且做出来的应用漂亮、能用。另一个是 Monologue 的新功能发布，他在预发布测试里用掉大约 9 亿 tokens。

"他用了大约 9 亿 tokens 测 GPT-5.5，并说这是他做任何事情时最喜欢的模型。"

Naveen 的评价很重：他认为自己能按时把 Monologue 新功能发出来，靠的就是这个模型。Every 团队把这类反馈看得很高，因为它来自真实产品线，不只是 benchmark。当任务已经拆成明确计划，GPT-5.5 的角色更像一名能长时间跑任务的执行工程师。

Dayline 的场景尤其典型：原生 iOS 和 Mac 应用对细节、状态、平台差异都有要求，不能只靠一个 demo 截图糊弄过去。Naveen 的反馈重点落在执行完整度上：它能按计划把功能一路做完。对赶发布节点的团队，这比一次漂亮回答更有价值。

这里也能看到 token 消耗的变化。9 亿 tokens 听起来夸张，但对持续跑任务的产品团队来说，模型已经从“问答成本”变成“工程产能成本”。如果它能帮一个 GM 在期限前交付功能，团队就会自然开始计算：哪些开发任务适合交给模型长跑，哪些任务仍需要人类架构师把关。

设计感、Ruby 和模糊需求仍交给 Opus

GPT-5.5 并没有全面赢。Cora 的负责人 Kieran Classen 用 LFT bench 测了更产品化的工程任务：做功能、处理前端、照顾设计和产品判断。结果是 5.5 表现不错，但设计导向任务里，Opus 4.7 的审美更稳。Mike Keller 也有类似观察：从一个欠明确的提示出发 vibe code 复杂应用，Opus 4.7 更容易一路做完。

"如果你在 vibe coding，或者想做高级工程师类型的工作，更明确的计划会让这个模型发挥得最好。"

语言栈上也有差异。Kieran 是 Ruby 重度用户，他认为 5.5 写出来的 Ruby 质量不够好；TypeScript 和 Swift 则更让团队满意。工程团队可以据此分工：结构化重构、TypeScript、Swift、清晰计划交给 5.5；审美判断、模糊产品探索、Rails 代码质量检查，仍保留 Opus 4.7 的位置。

LFT bench 更接近产品团队日常：需求并不总是写清楚，前端体验和交互取舍会不断冒出来。5.5 在这种场景下不差，但原稿给出的判断很克制：如果任务需要审美、产品 taste、从模糊目标里自己找路，Opus 4.7 依旧更可靠。

商务写作开始摆脱 GPT 味

写作部分的结论更克制。GPT-5.5 的人格感不如老版 Opus，尤其不如 Opus 4.6，但它很适合商务写作。主讲人用它写投资人更新，几乎一次生成就接近可发送状态。Every 的 staff writer Katie Parrot 长期主要用 Claude 写作，这次也开始在一些写作任务上转向 GPT 模型。

"它在模仿一种风格时不错，而且不会过度表演。"

Katie 和 Mike 都提到，5.5 的声音复制做得好，能模仿风格，又不会把风格演到夸张。这种“收着写”的能力，让它更适合投资人更新、团队备忘录、产品说明和对外邮件。如果要的是强人格、强观点的文章，Opus 仍有优势；如果要的是稳、准、少废话，5.5 更像企业内部每天会用的写作工具。

主讲人的投资人更新测试很有代表性。企业写作最怕两种结果：一种是空泛热情，另一种是模仿口吻过猛。GPT-5.5 的优势来自克制，能把已有材料整理成接近可发送的版本。创始人、PM、部门负责人更可能先在周报、更新、邮件和决策备忘录里用上它。

Katie 的反馈也值得放在这个背景里看。她过去一两年几乎只用 Claude 做写作任务，这次开始把部分工作转给 GPT 模型，说明 5.5 的变化不只是“会写”，而是能进入专业写作者的日常工具箱。它不靠强烈个性取胜，靠的是少犯风格错误、少制造尴尬句子。

桌面知识工作成了它的主场

知识工作是 OpenAI 过去几个月追得最快的地方。主讲人提到，搭配 Codex Desktop 使用时，GPT-5.5 已经成了很强的桌面 agent 体验。它能使用电脑上的应用，能浏览网页，能写 dashboard，也能做复杂数据分析。另一个强感知是速度：和 Opus 4.7 相比，5.5 明显更快，OpenAI 的算力优势在日常使用里能直接感受到。

"它很快、很强，能使用你电脑上的任何应用，也擅长浏览网页。"

但主讲人也扣了一分：为了让模型输出更好消化，5.5 在细节洞察上有时不如 Opus。做需要锐利判断的评估任务，比如给模型轨迹打分，他更信任 Opus 4.7。日常使用上，他几乎把 5.5 当成桌面主力；需要审阅、洞察和判断时，Opus 仍然值得保留。

这一段也解释了为什么用户体感会变化。慢模型再聪明，放进桌面 agent 工作流里也会让人中断；快模型哪怕少一点锐度，只要能稳定用浏览器、表格、代码和本地应用，就会成为默认入口。Every 的判断是，5.5 已经足够像一个日常主力，但还没有取代 Opus 的审稿和裁判位置。

主讲人最后的个人用法也很直白：移动端还会用 Claude，电脑上的 agentic 工作基本转向 GPT-5.5。原因很务实：它稳定、便宜、速度快，并且在连续几周里更少忘事、更少做蠢事。这个判断来自每天工作的摩擦感，也来自一次次等待、重试、修补失败后的选择。

写在最后

这期最可操作的结论很简单：不要只问哪个模型最强，先拆清楚谁规划、谁执行、谁审阅。把任务写成具体合同，让 GPT-5.5 长时间动手；把审美、洞察和模糊探索交给 Opus。工程团队的 AI 工作流，正在从“选一个模型”变成“编排一组模型”。
