# PRD：拾贝 iOS MVP

## 1. Overview

产品名：拾贝。

Slogan：每天捡起一枚知识贝壳。

### 1.1 Problem

用户每天会在公众号、小红书、播客、视频、文章、工作资料和 AI 对话中遇到大量有价值的信息，但这些内容大多只是被收藏、截图、转发或复制到备忘录。传统收藏工具解决了“保存在哪里”的问题，却很少帮助用户真正复习、理解和记住。

另一方面，传统背诵和刷题工具通常要求学习材料本身已经结构化，例如单词、题库或课程。用户日常收集到的文章、笔记、视频和方法论片段往往是非结构化内容，整理成本高，最终容易停留在“保存过”，而不是“记住了”和“会用了”。

### 1.2 Target Users

第一版面向有持续学习需求、但知识来源分散且缺少整理时间的个人用户，尤其是：

- 关注 AI、大模型、产品、商业、设计、增长等主题的学习者。
- 经常收藏公众号文章、小红书笔记、视频内容、播客摘要或课程片段的人。
- 希望把碎片阅读转化为长期记忆的人。
- 工作中需要持续吸收新概念、新方法、新案例的产品经理、设计师、运营、创业者和知识工作者。
- 已经习惯背单词类产品，能够接受短时间、多次、轻量测试复习方式的用户。

### 1.3 User Scenarios

- 用户在公众号看到一篇关于 AI Agent 的长文章，觉得有启发但当下没有时间整理，于是复制文章链接到拾贝。系统生成章节后通知用户，用户进入章节详情并开始复习。
- 用户在小红书或社群里看到一段产品方法论，复制文字到拾贝。系统提炼知识点并生成理解型题目，用户在通勤或碎片时间打开 App 快速刷几道题。
- 用户看完一个 AI 工具或商业案例视频，把视频链接添加到拾贝。系统尝试提取字幕、转写或简介内容，生成可复习章节；如果无法获取有效文本，则展示清楚的失败原因和重试入口。
- 用户已经添加了多篇内容，但不想被每日任务绑定。用户打开 App 后只看到一个清晰的复习入口，可以自由开始、退出、继续。

### 1.4 Product Summary

拾贝是一个面向个人碎片知识的 iOS 复习应用。用户可以把日常看到的文字、文章链接和视频链接添加到 App，系统自动提炼知识点、生成测试题，并按章节组织复习。

拾贝的核心价值不是再做一个收藏夹，而是把用户随机收集到的非结构化内容，转化为可复习、可测试、可追踪的知识点和题目。产品希望提供一种类似百词斩、扇贝背单词的轻量复习体验，但学习对象不是单词，而是用户自己收集的碎片知识。

### 1.5 MVP Focus

拾贝 iOS MVP 聚焦验证两件事：

1. 用户添加真实内容后，AI 是否能生成有用、可信、能检验理解的题目。
2. 用户是否愿意用题卡形式复习自己收集的知识。

因此，第一版的核心不是完整知识库、复杂管理工具或社区内容平台，而是“高质量题目驱动的自由复习”。用户打开 App 后，首页只承担开始或继续复习这一件事；章节管理、处理中、失败和历史内容放到次级页面。

题目进入复习池前不要求用户确认。AI 需要先做质量自检，只有通过自检的题目自动入池。用户之后可以对题目进行删除、反馈或标记为不合适，但 MVP 不设置入池前人工确认流程。

## 2. Goals

1. 让用户能在 iOS 上快速添加碎片知识，单次添加操作控制在 10 秒以内。
2. 将文字、文章链接、视频链接内容自动转化为可复习的章节、知识点和测试题。
3. 以题目质量为 MVP 最重要成功标准，题目必须能检验理解，而不只是复述原文。
4. 按知识点维护掌握状态和复习计划，而不是只按单道题做错题本。
5. 让用户能按自己的节奏随时进入复习，而不是被强制完成固定每日题量。

## 3. User Stories

### US-001：快速添加文字内容

**Description:** As a learner, I want to paste copied text into the app so that I can quickly turn useful fragments into reviewable knowledge.

**Acceptance Criteria:**

- [ ] 用户可以在添加页粘贴一段中文文本。
- [ ] 提交后，系统创建一条来源记录，状态显示为“处理中”。
- [ ] 系统能从文本中提取至少 1 个知识点。
- [ ] 如果内容不足以生成知识点，系统显示“内容不足，无法生成题目”的结果。
- [ ] UI 需要在浏览器或 iOS 原型中验证主要流程可完成。

### US-002：粘贴文章链接并提取正文

**Description:** As a learner, I want to paste an article link so that useful long-form content can become a review chapter.

**Acceptance Criteria:**

- [ ] 用户可以在添加页粘贴文章链接，例如公众号文章链接或网页文章链接。
- [ ] 系统保存链接、标题、抓取状态和提取文本。
- [ ] 如果无法抓取正文，系统显示失败状态，用户可进入章节详情页查看原因并重试。
- [ ] 正文提取成功后，系统为该来源创建一个章节。
- [ ] UI 需要在浏览器或 iOS 原型中验证主要流程可完成。

### US-003：粘贴视频链接并提取内容

**Description:** As a learner, I want to paste a video link so that useful knowledge from video content can become a review chapter.

**Acceptance Criteria:**

- [ ] 用户可以在添加页粘贴视频链接。
- [ ] 系统保存视频链接、标题、平台信息、处理状态和提取文本。
- [ ] 系统优先从字幕、转写文本或可获取的文本信息中提取内容。
- [ ] 如果无法获取视频文本，系统提示用户改为粘贴视频摘要、字幕或笔记。
- [ ] 文本提取成功后，系统为该视频创建一个章节。
- [ ] UI 需要在浏览器或 iOS 原型中验证主要流程可完成。

### US-004：按来源创建章节

**Description:** As a learner, I want each uploaded source to become a clear chapter so that I can review knowledge by article, video, or pasted note.

**Acceptance Criteria:**

- [ ] 每次有效添加默认创建一个章节。
- [ ] 文章链接、视频链接、粘贴文字都可以成为章节来源。
- [ ] 章节展示标题、来源类型、生成状态、知识点数量和题目数量。
- [ ] 章节生成完成后，用户可以从该章节开始复习。
- [ ] 后续版本允许 AI 建议合并同主题章节，但 MVP 不自动强制合并。

### US-005：生成完成后进入章节落地页

**Description:** As a learner, I want to tap a generation-complete notification and land on the generated chapter so that I can immediately decide whether to start reviewing.

**Acceptance Criteria:**

- [ ] 章节生成完成后，系统在 App 内展示“新生成”状态。
- [ ] 如果用户允许通知，系统在章节生成成功或失败时发送通知。
- [ ] 用户点击通知后进入对应章节详情页，而不是直接进入第一道题。
- [ ] 章节详情页第一屏必须展示章节标题、来源信息、生成结果、状态、开始复习按钮和知识点列表。
- [ ] 从生成完成通知进入的章节详情页只需要一个主按钮：“开始复习”。
- [ ] 用户点击“开始复习”后进入题卡复习页。

### US-006：自动生成高质量题目

**Description:** As a learner, I want the AI to generate useful questions automatically so that I do not need to organize or edit every item myself.

**Acceptance Criteria:**

- [ ] 每个有效来源至少尝试提炼 1 个知识点。
- [ ] 每个知识点最多生成 1 到 3 道题。
- [ ] 题目类型至少支持选择题、判断题、场景判断题；概念辨析用选择题承载。
- [ ] 每道题必须包含题干、答案、解释、来源引用和对应知识点。
- [ ] 系统自动过滤无法被来源支撑、答案模糊、过于简单或高度重复的题目。
- [ ] 通过质量检查的题目自动进入复习池，无需用户入池前确认。

### US-007：自由进入章节复习

**Description:** As a learner, I want to open the app and continue my current review or choose a chapter to review so that I can study at my own pace.

**Acceptance Criteria:**

- [ ] App 首页展示一个主复习卡片和“开始复习/继续复习”主按钮。
- [ ] 用户可以从任意章节开始复习。
- [ ] 章节复习展示题目总数、当前进度和当前题卡。
- [ ] 用户答题后立即看到对错反馈、简短解释和来源入口。
- [ ] 完成一题后，用户可以进入下一题。
- [ ] 系统不强制用户完成固定每日复习量。
- [ ] UI 需要在浏览器或 iOS 原型中验证主要流程可完成。

### US-008：按知识点调度复习

**Description:** As a learner, I want the app to track my mastery by knowledge point so that related questions can reinforce the same concept over time.

**Acceptance Criteria:**

- [ ] 系统为每个知识点维护学习状态。
- [ ] MVP 学习状态至少包括：新知识、答错、反复答错、已掌握。
- [ ] 用户答题后，系统更新对应知识点的状态。
- [ ] 复习队列根据章节、知识点状态和用户选择生成，而不是只根据单题错题生成。
- [ ] 答错或点击“不知道”的知识点会在更近的复习周期再次出现。

### US-009：反馈低质量题目

**Description:** As a learner, I want to mark a question as bad so that low-quality generated content does not keep appearing.

**Acceptance Criteria:**

- [ ] 用户可以在答题反馈页标记“题目不合适”。
- [ ] 被标记的题目不再进入默认复习队列。
- [ ] 系统记录低质量原因，至少包括：答案不准、太简单、题目看不懂、和来源无关。
- [ ] 低质量反馈可用于后续改进生成策略。

## 4. Functional Requirements

- FR-1: 系统必须提供 iOS 优先的产品体验，核心路径为添加内容、自动生成章节和题目、首页开始/继续复习、章节页查看所有章节。
- FR-2: 系统必须支持三种输入方式：粘贴文字、粘贴文章链接、粘贴视频链接。
- FR-3: 系统必须为每次输入创建来源记录，并保存来源类型、原始内容、提取文本、创建时间和处理状态。
- FR-3.1: 系统必须默认按来源创建章节；一次有效添加默认对应一个章节，后续版本允许 AI 建议合并同主题章节。
- FR-3.2: 系统必须在章节生成完成后提供明确落地页，用户可以从该页面直接开始复习。
- FR-3.3: MVP 中从生成完成通知进入的章节详情页只需要支持“已生成”状态，并提供“开始复习”主按钮。
- FR-4: 系统必须从来源内容中提炼知识点，每个知识点至少包含主题、摘要、关键判断和来源引用。
- FR-5: 系统必须优先生成能检验理解的题目，尤其是场景判断题和概念辨析类选择题。
- FR-6: 系统必须为每道题保存题型、题干、选项、正确答案、解释、来源引用、对应知识点和质量状态。
- FR-7: 系统必须在题目入池前进行自动质量检查。
- FR-8: 系统不得要求用户在每批题进入复习池前手动确认。
- FR-9: 系统必须允许用户在复习后删除或反馈低质量题目。
- FR-10: 系统必须按知识点维护掌握状态，并据此生成复习队列。
- FR-11: 系统必须在章节复习中展示进度和单题反馈。
- FR-12: 系统必须允许用户从题目回看来源片段、文章链接或视频链接。
- FR-13: 系统必须记录最近答错或反复答错的知识点，并把这些状态用于复习队列调度；MVP 不提供单独薄弱点视图。
- FR-14: 系统必须在处理失败时给出明确状态，例如文章抓取失败、视频文本提取失败、内容不足、题目生成失败。
- FR-15: 系统必须把 AI 内容处理流程拆分为可追踪阶段：内容清洗、语义分块、知识点候选提取、知识点筛选、题目生成、质量检查、入池。
- FR-16: 系统必须为每个知识点保存来源片段引用，确保题目答案可以回溯到原始内容。
- FR-17: 系统必须记录题目生成失败或被质量检查过滤的原因，用于后续调优。
- FR-18: 系统必须支持章节生成状态机，并将后台细状态映射为用户可理解的前台状态。
- FR-19: 系统必须在失败时提供对应操作；失败页按钮区只保留主要恢复动作，例如重试或重新生成，以及“不再提示”。如果已提取知识点，知识点列表底部使用“查看全部 X 个”入口，不再额外放置“查看知识点”按钮。来源通过来源信息链接进入。用户点击“重新生成”提交成功后返回首页，并展示提交成功浮窗。
- FR-20: 失败章节必须保留在章节页中；用户关闭提示后，不再出现在通知页中。
- FR-21: 系统必须在用户第一次添加成功后请求 iOS 通知权限，并说明通知用于提醒内容生成完成。
- FR-22: 系统必须在章节生成成功或失败时发送通知，前提是用户已允许通知。
- FR-23: 用户点击生成通知后，系统必须进入对应章节详情页或章节失败状态页。
- FR-24: 系统必须提供全部章节页，按最近创建时间展示所有章节。
- FR-25: 全部章节页 MVP 不支持搜索、筛选、标签、文件夹、批量管理或列表内删除。
- FR-26: 系统必须允许用户在章节详情页删除章节；删除章节必须二次确认，确认后永久删除。
- FR-27: 系统必须提供新用户空状态；首页空状态只展示一行提示文案，引导用户点击底部导航中间的 `+` 添加内容，不提供页面内按钮。
- FR-28: 系统必须说明用户上传内容仅用于生成知识点、题目和复习体验，不公开、不共享。
- FR-29: 系统必须提供底部导航；添加入口必须放在底部导航中间的 `+`，底部 tab 文案统一为“添加”。
- FR-30: 首页不得展示章节管理列表、失败处理列表或复杂推荐信息；这些内容应进入章节页或通知页。
- FR-31: 系统必须提供“完整知识点页”，用于从章节详情页或题目生成失败章节查看本章全部知识点；该页面只负责查看，不承担复习调度、筛选、收藏或编辑。
- FR-32: 通知页 MVP 不支持左滑已读、左滑删除、批量已读或批量删除；通知状态由用户点击进入、重新生成成功和“不再提示”等明确动作驱动。
- FR-33: MVP 必须支持用户账号，用于在服务器端保存用户上传内容、章节、知识点、题目和复习记录，并将数据绑定到具体用户。
- FR-34: AI 生成题目、知识点提取、正文提取、prompt、质量检查、模型调用和 API Key 管理必须部署在服务端；iOS App 不得内置生产生成逻辑或直接调用大模型。iOS 只负责输入、展示、复习交互、轻量本地状态和服务端 API 调用。

## 5. Non-Goals (Out of Scope)

第一版不做以下能力：

- Android 客户端。
- Web 完整产品。
- 微信、小红书、公众号等平台的深度 API 集成。
- 浏览器插件。
- 截图上传和 OCR 识别。
- 公共题库或内容社区。
- 团队协作。
- 订阅、会员等级、套餐页、付费墙和 Free Plan 展示。
- 复杂知识图谱。
- 重度笔记编辑器。
- 全部章节页搜索和筛选。
- 章节标签、文件夹、收藏、归档和批量管理。
- 通知左滑操作、通知删除、批量已读和通知批量管理。
- 单独薄弱点页、错题本页或知识点掌握度看板。
- 首页推荐系统或每日强制学习量。
- 以全文总结为主的阅读工具。
- 导出题库、组卷、打印等重学习工具能力。
- 用户入池前逐题审核流程。

## 6. AI Extraction and Question Generation Logic

这一部分是 MVP 的核心能力。系统不应该简单地“总结文章并出几道题”，而应该把用户上传内容转成一组可复习、可验证、可调度的知识点。

### 6.0 生成系统部署边界

AI 生成题目是拾贝的核心服务端能力，不应打包进 iOS App。生产形态中，iOS App 通过后端 API 提交文字、文章链接或视频链接；后端负责正文提取、内容清洗、知识点提取、题目生成、质量检查、失败重试、状态更新和通知触发。

这条边界的原因：

- API Key、模型配置和 prompt 不能放在客户端，避免被反编译或滥用。
- 题目生成策略、质量检查和模型选择会频繁迭代，应通过服务端发布，而不是依赖 App 发版。
- 生成请求涉及成本、限流、失败重试、风控和日志分析，必须由服务端统一治理。
- 文章抓取、视频文本提取和题目生成可能是长任务，适合由后端异步任务处理，并通过章节状态和通知反馈给 iOS。
- 未来账号同步、多端访问和历史复习记录都依赖服务端作为数据源。

iOS 端可以在 mock、离线演示或弱网兜底中保留轻量 ReviewSession 调度规则，但生产主状态以服务端为准。iOS 不承载 prompt、质量检查、正文提取、模型调用或生产级生成规则。

### 6.1 内容清洗

系统先把用户输入转成干净文本。

- 文字输入：去除多余空行、广告语、无意义表情、重复片段。
- 文章链接输入：提取标题、正文、发布时间或来源名；失败时展示失败状态，用户可查看原因并重试。
- 视频链接输入：优先提取标题、平台信息、字幕、转写文本或简介；失败时提示用户粘贴视频摘要、字幕或笔记。

清洗后的内容必须保留和原始来源的映射，便于后续展示“这道题来自哪里”。

### 6.1.1 章节创建

内容清洗成功后，系统默认为该来源创建一个章节。

章节创建规则：

- 一次粘贴文字默认创建一个章节。
- 一个文章链接默认创建一个章节。
- 一个视频链接默认创建一个章节。
- 如果一次输入包含多个链接，系统应提示用户拆分或为每个链接分别创建章节。
- 章节标题优先使用文章标题、视频标题或用户输入的首句摘要。

MVP 不自动强制合并章节。后续版本可以让 AI 识别同主题章节，并建议用户合并，例如把多篇 RAG 相关文章合并到一个“RAG”主题合集。

### 6.2 语义分块

系统需要把长内容切成适合提炼的语义块，而不是按固定字数粗暴切分。

每个语义块应该尽量表达一个完整观点，例如：

- 一个概念定义
- 一个原因解释
- 一个方法步骤
- 一个适用场景
- 一个反例或误区
- 一个观点和论据组合

如果内容很短，可以只有一个语义块。如果内容很长，系统需要保留块之间的上下文关系，例如“前文定义了概念，后文给出案例”。

### 6.3 知识点候选提取

系统从每个语义块里提取候选知识点。候选知识点不是原文摘抄，而是更适合记忆和测试的表达。

每个候选知识点应包含：

- 主题：例如 RAG、产品定位、增长模型。
- 知识类型：概念、判断、方法、场景、反例、对比、步骤。
- 核心表述：一句话说明这个知识点。
- 来源片段：支持该知识点的原文片段。
- 可考性评分：判断这个知识点是否适合生成题目。

示例：

原文：RAG 可以缓解模型知识更新不及时的问题，但它不能让模型真的拥有长期记忆。

候选知识点：

- 主题：RAG
- 知识类型：概念辨析
- 核心表述：RAG 适合补充外部知识和事实检索，但不等于模型拥有长期记忆。
- 来源片段：原文对应句子
- 可考性评分：高

### 6.4 知识点筛选

不是所有内容都应该变成题。系统需要过滤掉低价值候选。

应优先保留：

- 能帮助用户形成判断的观点
- 容易混淆的概念
- 有明确适用场景的方法
- 常见误区或反例
- 可以迁移到实际工作中的原则

应过滤：

- 纯情绪表达
- 缺少上下文的孤立句子
- 无法被来源支持的推断
- 过度宽泛的常识
- 太细碎、记住也没有价值的信息

### 6.5 题型选择策略

系统应该根据知识点类型选择题型。

- 概念类：优先生成选择题。
- 判断类：优先生成判断题，但解释必须说明为什么。
- 方法类：优先生成步骤题或场景应用题。
- 场景类：优先生成场景判断题。
- 反例/误区类：优先生成“以下哪种理解是错误的”。
- 对比类：优先生成“二者区别是什么”的选择题。

题型选择的目标不是多样化本身，而是让题目最能检验该知识点。

MVP 支持三种题型：

```text
multiple_choice
true_false
scenario_judgment
```

概念辨析题不作为独立题型。需要概念辨析时，用选择题承载。

判断题的固定选项文案为：

```text
成立 / 不成立
```

场景判断题统一使用 `scenario_judgment` 类型，但可以有不同形态：

- 场景选择：给一个场景，问最合适的做法。
- 场景是否适用：判断某方法是否适合该场景。
- 方案对比：比较两个方案哪个更符合知识点。

前端不需要为场景题做特殊组件，统一根据 `options` 渲染。

### 6.6 题目生成规则

每个知识点最多生成 1 到 3 道题。第一版可以默认生成 1 道主题；只有当知识点足够重要或容易混淆时，才生成多道题。

每道题必须包含：

- 题干
- 题型
- 选项，如果题型需要
- 正确答案
- 答案解释
- 正确理解
- 常见误区
- 来源引用
- 对应知识点 ID
- 难度
- 质量评分

题干应避免直接复述原文。优先使用这些方式：

- 换一个场景问用户能否应用。
- 给出一个错误理解，让用户识别。
- 对比两个相近概念。
- 问“为什么不是另一个答案”。

### 6.6.1 统一解释页字段

统一解释页由 Question 和 ReviewAttempt 共同渲染。

Question 需要提供：

```text
correct_answer
explanation
correct_understanding
common_misconception
source_snippet
```

ReviewAttempt 需要提供：

```text
user_answer
result
```

解释页展示规则：

- 解释页不展示“你的答案”。
- 解释页展示“正确答案”和“正确理解”。
- “正确理解”承担核心解释职责，不单独展示“为什么对”。
- 点击“不知道”时，不需要在解释页重复展示“不知道”。
- MVP 不解释每个错误选项为什么错。
- MVP 不为每个错误选项单独生成解释字段。

解释页结构：

```text
正确答案
正确理解
常见误区
来源片段
```

### 6.7 质量检查

题目进入复习池前，系统必须做自动质量检查。

每道题至少检查：

- 来源支撑：答案能否从来源片段中得到支持。
- 答案唯一性：是否只有一个最合理答案。
- 理解深度：是否不只是原文填空或机械记忆。
- 表达清晰：题干和选项是否没有歧义。
- 干扰项质量：错误选项是否合理但明确错误。
- 重复度：是否和已有题目高度重复。

质量检查结果可以分为：

- 通过：自动进入复习池。
- 重写：返回题目生成阶段重写一次。
- 丢弃：不进入复习池，并记录原因。

### 6.8 知识点级复习调度

复习调度的对象是知识点，题目是测试知识点的载体。

当用户答对某道题时，系统更新对应知识点的掌握状态。当用户答错时，系统降低该知识点状态，并优先安排相关题目再次出现。

如果一个知识点有多道题，系统应该避免短时间内重复刷到同一表述，而是换一种问法测试同一知识点。

### 6.8.1 复习流动态推荐

复习推荐系统只发生在用户进入章节复习之后，用于决定复习过程中的下一题或下一个知识点。MVP 不在首页做复杂推荐；首页只展示一个可开始或可继续的主复习入口。

推荐对象是知识点，不是章节。题目是知识点的测试载体。

#### 掌握分

每个知识点维护一个 `mastery_score`：

```text
mastery_score: 0-100
初始值：50
```

前台不需要展示这个分数。它只用于后台排序和复习流推荐。

#### 新知识点定义

一个知识点如果从未被用户作答，则视为 `new`。

规则：

- 知识点进入过队列但用户中途退出，仍然是 new。
- 用户答过一次，不管对错，都不再是 new。
- new 知识点在章节复习中优先出现。

#### 章节复习队列生成

用户点击某章节的“开始复习”后，系统生成本轮章节复习队列。

规则：

```text
1. 一轮章节复习默认覆盖本章全部知识点。
2. new 知识点优先进入队列。
3. 非 new 知识点按 mastery_score 从低到高排序。
4. 每个知识点第一次出现时，选择 quality_score 最高的题目。
5. 如果 quality_score 相同，优先选择场景判断题或概念辨析类选择题。
```

如果章节是全新章节，所有知识点都是 new，系统可以按 AI 生成顺序或知识点顺序出题。

#### 答题后的分数更新

MVP 使用简单可解释的分数更新规则：

```text
答对：+15
答错：-20
点击“不知道”：-20
强化答对：+10
强化仍错：-15，并再次进入强化队列
强化仍点击“不知道”：-15，并再次进入强化队列
题目有问题：不更新 mastery_score
```

分数更新后必须限制在 0 到 100 之间。

如果用户反馈“题目有问题”，系统不应把这次作答用于判断用户是否掌握该知识点。

#### 本轮强化队列

答错或点击“不知道”的知识点进入本轮强化队列。MVP 中，章节复习需要把本章知识点都刷到本轮答对，才算完成。用户可以中途退出，但退出时该 ReviewSession 保持未完成，下次继续。

强化规则：

```text
1. 强化题固定间隔 3 道其他题后出现。
2. 如果本轮剩余未答知识点不足 3 个，强化题排到本轮末尾。
3. 同一知识点会持续进入强化队列，直到用户在本轮答对。
4. 强化时优先选择同一知识点下另一道题。
5. 如果没有另一道题，可以重复同题。
6. 如果用户中途退出，本轮未答对的强化知识点保留在未完成 session 中。
```

强化结果：

```text
强化答对：
  -> 记录为本轮已掌握
  -> mastery_score +10
  -> 本轮不再主动出现

强化仍错或仍点击“不知道”：
  -> mastery_score -15
  -> 再次进入强化队列
  -> 固定间隔 3 道其他题后再次出现；如果剩余题不足 3 道，则排到本轮末尾
```

#### ReviewSession 完成条件

一轮章节复习完成需要同时满足：

```text
1. 本章所有知识点都至少被用户作答一次。
2. 本轮中所有答错或点击“不知道”的知识点，后续都已强化到答对。
3. 当前强化队列为空。
```

如果用户中途退出，ReviewSession 不完成。下次用户点击该章节的“继续复习”时，系统恢复未完成队列和强化队列。

章节完成页不展示“未掌握知识点”。因为在 MVP 规则下，只要进入完成页，就代表本章知识点已经在本轮全部答对。

章节完成页应复用章节详情页的信息结构，并展示动作按钮：`继续下一章`，如果存在下一章；以及 `回到章节`。如果来源是文章链接或视频链接，页面必须展示来源链接入口。

完成页示例：

```text
┌──────────────────────────────┐
│ < 章节总结                     │
├──────────────────────────────┤
│ 如何把 AI Agent 用进你的生意、 │
│ 内容和生活                     │
│                              │
│ 来源：公众号文章               │
│ 账号：晚点再听LaterCast        │
│ 链接：mp.weixin.qq.com/s/...   │
│                              │
│ 12 个知识点 · 18 道题          │
│ 状态：本章复习完成             │
│                              │
│ [继续下一章]                  │
│ [回到章节]                    │
├──────────────────────────────┤
│ 本章知识点                     │
│                              │
│ 1. 非技术人使用 AI 的核心能力  │
│ 2. 不要把 AI 只当搜索框        │
│ 3. 先写清目标，再打开工具      │
│ 4. Agent 的关键是采取行动      │
│ 5. 先找瓶颈，再谈自动化        │
│ 6. 从低风险任务开始建立信任    │
│                              │
└──────────────────────────────┘
```

#### 中途退出与继续复习

用户可以随时退出复习，不弹确认。系统自动保存当前 ReviewSession。

退出恢复规则：

```text
题卡页未作答退出：
  -> 下次继续时回到同一道题
  -> 不记录作答，不更新 mastery_score

解释页退出，且还没点“继续复习”：
  -> 下次继续时回到该解释页

解释页已点“继续复习”后退出：
  -> 下次继续时进入下一题

章节总结页退出：
  -> ReviewSession 已完成
  -> 下次进入该章节时显示章节详情页
```

首页主复习卡片：

```text
1. 如果存在未完成 ReviewSession，展示最近复习的未完成章节。
2. 如果没有未完成 ReviewSession，但存在已生成未复习章节，展示最近生成的可开始章节。
3. 首页只展示一个章节卡片。
4. 其他章节从“章节”页进入。
```

章节详情页主按钮：

```text
没有未完成 ReviewSession：开始复习
有未完成 ReviewSession：继续复习
```

章节总结页的“继续下一章”规则：

```text
下一章 = 最近创建且还没有完成过复习的章节
如果存在下一章，显示“继续下一章”
如果不存在下一章，不显示该按钮
```

“继续下一章”不是首页推荐系统，只是章节完成后的轻量衔接。

#### 时间衰减

为了模拟遗忘，MVP 使用简单时间衰减，而不是复杂记忆曲线。

规则：

```text
每个知识点每天未复习，mastery_score -3
最低不低于 0
同一天只衰减一次
```

系统需要记录：

```text
last_reviewed_at
last_decay_applied_at
```

衰减可以在生成章节复习队列时计算，也可以由后台定时任务计算。MVP 优先选择在生成复习队列时按需计算，减少后台复杂度。

### 6.9 用户反馈闭环

用户不需要在入池前确认题目，但复习时可以反馈题目质量。

反馈类型包括：

- 答案不准
- 太简单
- 题干看不懂
- 和来源无关
- 已经掌握，不想再看

系统收到反馈后应更新题目质量状态。严重问题题目不再进入默认复习队列。

### 6.9.1 题目有问题反馈处理

统一解释页提供“题目有问题”入口。点击后展示底部弹窗：

```text
这道题哪里有问题？

[答案不准]
[题目看不懂]
[和来源无关]
[太简单]
[取消]
```

MVP 不提供“其他”输入框，避免增加审核复杂度。

反馈类型分为严重反馈和轻反馈。

严重反馈：

```text
答案不准
题目看不懂
和来源无关
```

严重反馈处理规则：

```text
1. 撤销本次作答对 mastery_score 的影响。
2. 当前题退出默认复习池。
3. 当前题本轮不再出现。
4. 如果该知识点还有其他可用题，后续优先换题。
5. 如果该知识点没有其他可用题，本轮直接跳过该知识点。
6. 因严重题目反馈被跳过的知识点，不计入本轮完成条件。
```

撤销分数时，使用 `ReviewAttempt.mastery_score_before` 恢复知识点分数。

轻反馈：

```text
太简单
```

轻反馈处理规则：

```text
1. 不撤销 mastery_score。
2. 不全局下架题目。
3. 该题对当前用户降权。
4. 后续优先选择该知识点的其他题。
```

反馈后显示轻提示。不同反馈类型使用不同文案，让用户知道反馈已经生效：

```text
答案不准 / 题目看不懂 / 和来源无关：
已收到，这道题已从本次复习移除

太简单：
已收到，后续会减少出现
```

然后用户继续复习。

### 6.10 失败和降级逻辑

AI 处理失败时，系统需要给用户明确结果，而不是静默失败。

常见失败包括：

- 文章链接无法抓取正文。
- 视频链接无法提取字幕、转写文本或可用摘要。
- 内容太短，无法提炼知识点。
- 内容可以总结，但不适合出题。
- 题目生成成功，但连续两次都没有任何题目通过质量检查。

这些情况下，系统应展示具体原因，并给出下一步建议，例如“请稍后重试”或通过来源链接查看原始内容。

### 6.11 章节生成状态机

后台需要保留细粒度状态，方便排查问题和给前台展示正确阶段。前台不需要展示完整技术流程，但需要显示当前正在做什么。

状态映射：

```text
后台状态                  前台显示
--------------------------------------------
submitted                 已提交，等待处理
extracting_content         正在提取正文
generating_points          正在生成知识点
generating_questions       正在生成题目
quality_checking           正在检查题目质量
auto_regenerating_questions 正在检查题目质量
completed                  已生成
failed_extract_article     文章正文提取失败
failed_extract_video       视频文本提取失败
failed_points              知识点生成失败
failed_questions           题目生成失败
failed_no_qualified_questions 题目生成失败
```

章节页的处理中卡片应使用阶段状态，例如：

```text
正在提取正文
正在生成知识点
正在生成题目
正在检查题目质量
```

### 6.12 失败处理

不同失败原因需要对应不同操作。MVP 不把“题目第一次质量检查没过”直接暴露给用户；系统应先自动恢复一次，避免用半成品状态打扰用户。

题目质量检查自动恢复规则：

```text
第一次题目质量检查：0 道题通过
  -> 系统自动重新生成一次题目
  -> 不通知用户
  -> 不进入失败页
  -> 保留已提取知识点

第二次题目质量检查：仍然 0 道题通过
  -> 标记为 failed_no_qualified_questions
  -> 如果用户允许通知，发送生成失败通知
  -> 用户点击通知进入章节详情页的失败状态
  -> 用户可手动点击“重新生成”
```

用户手动点击“重新生成”时，优先复用已提取知识点，只重新跑题目生成和质量检查；除非来源内容提取或知识点本身失败，否则不重新抓取原文或重新生成知识点。

处理规则：

```text
failed_extract_article
  -> 显示：文章正文提取失败
  -> 操作：[重试] [不再提示]

failed_extract_video
  -> 显示：视频文本提取失败
  -> 操作：[重试] [不再提示]

failed_points
  -> 显示：知识点生成失败
  -> 操作：[重试] [不再提示]

failed_questions
  -> 显示：题目生成失败
  -> 操作：[重新生成] [不再提示]
  -> 如果已提取知识点，在知识点列表底部展示“查看全部 X 个”

failed_no_qualified_questions
  -> 显示：生成失败
  -> 操作：[重新生成] [不再提示]
  -> 如果已提取知识点，在知识点列表底部展示“查看全部 X 个”
```

当系统已经生成知识点但连续两次都没有可用题目时，该章节不进入复习状态，不能点击“开始复习”。用户可以通过知识点列表底部的“查看全部 X 个”查看知识点，或手动重新生成题目。

失败章节仍然保留在全部章节中，避免用户上传内容丢失。来源入口不作为按钮出现在失败页按钮区，而是作为来源信息中的可点击链接展示。

## 7. Question Quality Standards and Evaluation

题目质量是 MVP 的第一成功标准。系统必须明确“好题”和“差题”的判断标准，并把这些标准用于 AI 自检、人工抽样评估和用户反馈闭环。

### 7.1 通用质量评分维度

每道题生成后，系统需要按以下维度进行质量评分。第一版可以使用 1 到 5 分制。

#### 7.1.1 来源支撑性

衡量题目的答案和解释是否能被原始来源支持。

- 5 分：答案和解释都能直接从来源片段中得到支持。
- 3 分：答案大体符合来源，但解释需要少量合理推断。
- 1 分：答案依赖来源之外的信息，或来源无法支持。

低于 4 分的题目不应直接进入复习池。

#### 7.1.2 答案唯一性

衡量题目是否只有一个最合理答案。

- 5 分：正确答案明确，其他选项明显不成立。
- 3 分：正确答案相对更好，但存在容易争议的选项。
- 1 分：多个选项都可能正确，或题目本身无法判断。

低于 4 分的题目必须重写或丢弃。

#### 7.1.3 理解深度

衡量题目是否能测试用户理解，而不只是测试机械记忆。

- 5 分：要求用户识别适用场景、误区、因果关系、概念边界或迁移应用。
- 3 分：主要测试定义或结论，但仍有一定理解要求。
- 1 分：几乎只是原文填空、关键词复述或常识判断。

MVP 应优先提高这一项。理解深度低的题目可以进入补充题，但不应成为主要题目。

#### 7.1.4 表达清晰度

衡量题干、选项和解释是否清楚。

- 5 分：题干明确，选项互斥，解释简洁。
- 3 分：基本可懂，但表述略绕或选项边界不够清晰。
- 1 分：题干含糊、选项重复、解释无法帮助理解。

低于 4 分的题目需要重写。

#### 7.1.5 干扰项质量

适用于选择题。衡量错误选项是否合理但明确错误，尤其是概念辨析类选择题。

- 5 分：错误选项来自常见误区或相近概念，能有效检验理解。
- 3 分：错误选项可用，但有些太明显。
- 1 分：错误选项荒谬、无关或明显凑数。

干扰项质量低会让题目变成“送分题”，应优先重写。

#### 7.1.6 复习价值

衡量这道题是否值得用户反复复习。

- 5 分：对应知识点重要、可迁移、容易被用到或容易混淆。
- 3 分：有一定价值，但更像细节记忆。
- 1 分：记住也没有明显帮助，或内容太碎。

复习价值低于 3 分的题目应丢弃。

### 7.2 入池规则

每道题经过质量评分后，系统给出一个处理结果。

- 通过：进入复习池。
- 重写：回到题目生成阶段，最多重写 1 次。
- 丢弃：不进入复习池，记录原因。

建议第一版入池规则：

- 来源支撑性 >= 4
- 答案唯一性 >= 4
- 表达清晰度 >= 4
- 复习价值 >= 3
- 总体平均分 >= 4

如果题目理解深度低于 3，但其他分数合格，系统可以降低其优先级，而不是直接丢弃。

### 7.3 题型专项标准

不同题型的好题标准不同，系统需要按题型做额外检查。

#### 7.3.1 选择题

好选择题应该：

- 只有一个明确正确答案。
- 错误选项来自相近概念、常见误区或错误应用场景。
- 不能通过明显长度、语气或绝对词猜答案。
- 解释需要说明为什么正确选项对，最好也说明为什么关键错误选项不对。

差选择题包括：

- 错误选项明显无关。
- 多个选项都能说得通。
- 题干只是问“原文提到了什么”。
- 答案可以靠排除法秒猜。

#### 7.3.2 判断题

好判断题应该：

- 测试一个明确判断，而不是模糊观点。
- 错误判断应该接近常见误解。
- 解释必须说明判断成立或不成立的边界。

差判断题包括：

- “是不是有用”“是否重要”这类过于主观的问题。
- 只要看关键词就能判断。
- 没有解释边界，用户答对也不一定理解。

#### 7.3.3 场景判断题

好场景判断题应该：

- 给出具体场景和目标。
- 要求用户判断某个方法、概念或原则是否适用。
- 场景必须和来源知识点有清晰关系。
- 解释要指出适用或不适用的关键条件。

差场景判断题包括：

- 场景太空泛，例如“某公司想增长，应该怎么做”。
- 场景和知识点关系牵强。
- 答案依赖大量来源之外的假设。

#### 7.3.4 概念辨析类选择题

好概念辨析类选择题应该：

- 比较两个容易混淆的概念、方法或说法。
- 明确考察边界、适用场景或关键差异。
- 解释要帮助用户建立“什么时候用 A，什么时候用 B”的判断。

差概念辨析类选择题包括：

- 只是问两个概念的字面定义。
- 比较对象并不容易混淆。
- 差异解释过于抽象，无法指导实际判断。

### 7.4 人工抽样评估

MVP 阶段需要人工抽样检查题目质量，避免只相信 AI 自评分。

建议流程：

1. 每周随机抽样 50 到 100 道新生成题目。
2. 按 7.1 的六个维度打分。
3. 记录低质量原因。
4. 统计各题型通过率。
5. 根据失败原因调整知识点筛选和题目生成策略。

人工评估的核心指标：

- 可用题比例：人工认为可以直接进入复习的题目比例。
- 高质量题比例：平均分 >= 4.5 的题目比例。
- 严重问题比例：来源不支撑、答案不唯一、解释错误等问题的比例。
- 低理解深度比例：只是复述原文或考关键词的题目比例。

### 7.5 固定测试集

MVP 不允许只靠主观体验判断题目生成质量。进入实现前，必须建立一组固定题目质量测试集，用于验证知识点提取、题目生成、解释生成和质量检查链路。

PRD 阶段先确定测试集要求；具体样本收集、标注表、评分脚本和执行流程在 write plan 阶段拆成任务。

第一版测试集建议包含 10 到 20 条代表性内容：

- 公众号文章。
- 普通网页文章。
- 视频链接或视频字幕文本。
- 用户手动粘贴的短文本。
- 产品理论、AI/大模型知识、方法论类内容。

每条测试内容至少记录：

- 原始内容或来源链接。
- 内容类型。
- 人工认可的核心知识点，建议 5 到 15 个。
- 题目生成结果。
- 每道题对应的知识点。
- 每道题的来源支撑片段。
- 按 7.1 六个维度给出的人工评分。
- 不通过原因。

测试集的用途：

```text
1. 验证 AI 是否能提取出真正值得复习的知识点。
2. 验证题目是否能检验理解，而不只是复述原文。
3. 验证答案和解释是否能被来源支撑。
4. 验证不同内容类型下的失败模式。
5. 每次调整 prompt、模型、知识点筛选或题型策略后做回归评估。
```

MVP 上线前，测试集结果应作为上线门槛。建议第一版门槛：

- 可用题比例 >= 70%。
- 严重问题比例 <= 10%。
- 来源支撑性平均分 >= 4。
- 答案唯一性平均分 >= 4。
- 表达清晰度平均分 >= 4。

这些门槛是第一版建议值，后续可根据测试集结果调整。

### 7.6 用户反馈闭环

用户反馈是题目质量的长期改进信号。

当用户标记“题目不合适”时，系统需要记录：

- 题目 ID
- 知识点 ID
- 来源 ID
- 用户选择的原因
- 用户是否答错
- 反馈发生的复习阶段

处理规则：

- “答案不准”：题目立即退出默认复习队列，进入待复查状态。
- “和来源无关”：题目立即退出默认复习队列，并降低该来源片段的生成置信度。
- “太简单”：题目可以保留，但降低复习优先级。
- “题干看不懂”：题目进入重写候选。
- “已经掌握，不想再看”：更新知识点状态，而不是判断题目质量差。

### 7.7 题目质量仪表盘

即使第一版不做复杂后台，也需要在数据层保留质量分析能力。

至少需要能统计：

- 每个来源生成了多少知识点和题目。
- 多少题通过、重写、丢弃。
- 每类丢弃原因的数量。
- 每种题型的用户反馈率。
- 每个知识领域的低质量题比例。

这些数据用于判断问题出在内容解析、知识点提炼、题型选择，还是题目表达。

## 8. Design Considerations

本章节中的示例页面使用以下真实文章作为样例来源：

- 标题：如何把 AI Agent 用进你的生意、内容和生活丨Calum Johnson
- 来源：晚点再听LaterCast
- 类型：公众号文章
- 核心内容：非技术人如何把 AI Agent 从问答工具用进真实工作流，包括任务描述、边界设定、瓶颈识别、低风险自动化和权限控制。

### 8.1 首页

首页应为“单一复习行动首页”，而不是章节管理页、资料库或添加页。用户打开 App 后，首页只回答一个问题：现在要不要开始复习。

首页不展示复杂章节列表，不展示失败处理列表，不展示推荐复习信息。处理中、失败、新生成等章节状态进入“章节”页或“通知”页。

首页应展示：

- 当前可继续或可开始的一个章节卡片
- 一个主按钮：开始复习或继续复习
- 一个轻量累计指标，例如已复习知识点数量或连续复习天数
- 底部导航，中间为 `+` 添加入口

如果用户还没有任何章节，首页展示空状态：

```text
┌──────────────────────────────┐
│ 复习                         │
├──────────────────────────────┤
│                              │
│     每天捡起一枚知识贝壳       │
│                              │
│   点击底部 + 添加复习内容       │
│ 支持文章/视频链接或粘贴文字    │
│                              │
├──────────────────────────────┤
│  首页      章节      + 添加  通知   我的 │
└──────────────────────────────┘
```

示例首页结构：

```text
┌──────────────────────────────┐
│ 复习                         │
├──────────────────────────────┤
│                              │
│          已复习知识点          │
│              35              │
│                              │
│ ┌──────────────────────────┐ │
│ │ 当前章节                  │ │
│ │ 如何把 AI Agent 用进...   │ │
│ │ 公众号文章                 │ │
│ │ 已复习 4/12 个知识点      │ │
│ └──────────────────────────┘ │
│                              │
│        [继续复习]              │
│                              │
├──────────────────────────────┤
│  首页      章节      + 添加  通知   我的 │
└──────────────────────────────┘
```

如果没有未完成章节，但存在已生成未复习章节，主卡片展示最近生成的可开始章节，按钮为“开始复习”。

底部导航规则：

```text
首页：开始/继续复习
章节：全部章节、处理中、失败
+ 添加：添加知识
通知：生成成功、生成失败
我的：设置、隐私说明、通知权限
```

### 8.1.1 文案规范

MVP 开发时使用以下最终文案口径：

- 产品名：拾贝。
- Slogan：每天捡起一枚知识贝壳。
- 底部导航：`首页 / 章节 / + 添加 / 通知 / 我的`。
- 中间 `+` 的 tab 文案使用“添加”，不使用“投喂”。
- 添加页标题：添加知识。
- 首页空状态文案：每天捡起一枚知识贝壳；点击底部 + 添加复习内容；支持文章/视频链接或粘贴文字。
- 添加页隐私提示：内容仅用于生成复习，不会公开。
- 我的页条目：账号信息、通知权限、隐私说明、关于拾贝、退出登录。
- MVP 需要账号用于云端保存数据，但不展示 Free Plan、订阅、会员等级、套餐或付费相关文案。

### 8.2 添加入口

添加入口应像聊天或快速记录入口。用户可以快速粘贴文本、粘贴文章链接、粘贴视频链接，不需要先选择分类。

提交后不应让用户停留在处理页。因为文章链接和视频链接处理可能较慢，提交成功后应立即返回首页，并在首页背景上展示提交成功浮窗。用户关闭浮窗后，首页不展示处理中状态；处理进度只在章节页或通知页查看。

用户第一次添加成功后，App 应请求 iOS 通知权限。请求前先展示自定义说明：

```text
我们会在内容生成好后通知你，方便你回来复习。

[开启通知]
```

用户点击“开启通知”后，再触发 iOS 系统权限弹窗。如果用户拒绝，系统仍可在章节页显示处理状态，并在 App 内通知页展示生成结果。

章节生成成功或失败都应发送通知，前提是用户已允许通知。

成功通知文案：

```text
《如何把 AI Agent 用进...》已生成 12 个知识点，点击开始复习。
```

失败通知文案：

```text
《如何把 AI Agent 用进...》生成失败，点击查看原因。
```

点击成功通知后进入对应章节详情页。点击失败通知后进入该章节的失败状态页或章节详情页中的失败状态。

示例添加流程：

```text
┌──────────────────────────────┐
│ 添加知识                       │
├──────────────────────────────┤
│ 粘贴文字 / 文章链接 / 视频链接 │
│                              │
│ https://mp.weixin.qq.com/s/...│
│                              │
│ [开始生成]                    │
└──────────────────────────────┘

提交后立即回到首页，并在首页背景上展示提交成功浮窗。首页本体保持简洁，不展示处理中列表；用户可以从“章节”或“通知”页查看处理进度：

┌──────────────────────────────┐
│ 复习                         │
├──────────────────────────────┤
│                              │
│      ┌────────────────┐      │
│      │ 已提交，正在生成 │      │
│      │ 完成后会通知你   │      │
│      │                │      │
│      │    [知道了]     │      │
│      └────────────────┘      │
│                              │
├──────────────────────────────┤
│  首页      章节      + 添加  通知   我的 │
└──────────────────────────────┘
```

### 8.3 章节详情页

章节详情页是 MVP 的核心页面之一。每个章节代表一次有效添加的内容来源，例如一篇文章、一个视频链接或一段粘贴文字。

章节详情页同时承担“生成完成通知落地页”的作用。用户从通知进入时，不应直接进入第一道题，而应先看到该章节已经生成了什么，并能在第一屏直接点击“开始复习”。

章节详情页也承担“生成失败通知落地页”的作用。用户点击失败通知后进入同一个章节详情结构，只替换状态区和按钮，不新建复杂失败页面。

章节详情页需要展示：

- 章节标题
- 来源类型
- 来源账号或平台
- 生成状态
- 知识点数量
- 题目数量
- 开始复习按钮
- 来源入口
- 知识点列表

知识点列表用于帮助用户理解“这一章有什么”，但不应做成复杂知识库。第一版不需要支持复杂编辑、拖拽排序或知识图谱。

章节详情页默认只展示部分知识点，避免页面过长。如果知识点数量超过 6 个，应在列表底部展示“查看全部 X 个”入口，进入完整知识点页。

完整知识点页规则：

- 只用于查看本章全部知识点。
- 每个知识点展示标题和一句话解释。
- 不展示掌握度、错题数、题目数等学习压力信息。
- 不支持搜索、筛选、收藏、编辑、排序。
- 点击单个知识点不进入详情页，MVP 保持纯列表。
- 顶部返回后回到章节详情页。

生成完成通知点击后的章节详情页示例：

### 8.4 题卡复习页

题卡应轻量、清晰、适合单手快速操作。每题答完后给出简短解释，详细来源通过入口展开，不应打断快刷节奏。

复习页以一道题一张卡片为核心。用户可以选择答案，也可以点击“不知道”。MVP 中“不知道”按答错处理，不单独维护“模糊”状态。

复习反馈规则：

- 答对旧知识点：题卡原地轻反馈，用户点击“下一题”后再进入下一题。
- 新知识点第一次出现：进入统一解释页。
- 答错：进入统一解释页。
- 点击“不知道”：进入统一解释页。

- 不显示“题目有问题”入口，避免干扰快刷节奏。

### 8.5 统一解释页

统一解释页用于新知识点、答错题和“不知道”题。MVP 不区分学习卡片和纠错卡片，避免生成逻辑和页面状态过复杂。

- 正确答案
- 正确理解
- 常见误区
- 来源片段
- 查看完整来源入口
- 题目有问题入口

### 8.6 来源详情页

来源详情页用于让用户确认题目和解释来自哪里。MVP 不需要还原复杂网页排版，只需要展示来源标题、来源类型、账号或平台、原文片段和打开原文入口。

来源入口规则：

- 如果来源是文章或视频链接，章节详情页的信息区展示可点击链接，点击后进入来源详情页；来源详情页再提供“打开原文链接”。

### 8.9 全部章节页

全部章节页是章节的兜底列表，不是复杂知识管理中心。MVP 保持极简。

页面规则：

- 按最近创建时间排序。
- 展示所有章节。
- 每个章节卡片展示标题、来源类型、状态、知识点数和题目数。
- 点击章节卡片进入章节详情页。
- 不支持搜索。
- 不支持筛选。
- 不支持标签、文件夹、收藏、归档。
- 不支持列表内删除或批量管理。
- 失败章节只显示状态，不在列表卡片上显示失败原因。

## 9. 页面地图与用户流程

本章节集中整理 MVP 的页面图、页面状态和跳转逻辑。页面原型以 ASCII 表达，用于在进入视觉稿或可点击原型前统一信息架构。

设计原则：

- 首页只承担开始/继续复习，不承担章节管理。
- 底部导航保持稳定：`首页 / 章节 / + 添加 / 通知 / 我的`。
- `+ 添加` 是添加知识入口，固定在底部导航中间。
- 章节状态、失败、处理中进入“章节”或“通知”。
- 复习中隐藏底部导航，保证专注。
- 所有错误和失败都要给出用户可理解的下一步。

### 9.1 全局导航

```text
┌──────────────────────────────┐
│          当前页面内容          │
│                              │
├──────────────────────────────┤
│  首页      章节      + 添加  通知   我的 │
└──────────────────────────────┘
```

导航职责：

```text
首页：开始/继续复习
章节：全部章节、处理中、失败
+ 添加：添加知识
通知：生成成功、生成失败
我的：设置、隐私说明、通知权限
```

#### 9.3.1 首页空状态

```text

│                              │

```

跳转：

```text
底部 [+] -> 添加知识页
```

#### 9.3.2 首页可复习状态

```text
┌──────────────────────────────┐
│ 复习                         │
├──────────────────────────────┤
│                              │
│          已复习知识点          │
│              35              │
│                              │
│ ┌──────────────────────────┐ │
│ │ 当前章节                  │ │
│ │ 如何把 AI Agent 用进...   │ │

│                              │
├──────────────────────────────┤
│  首页      章节      + 添加  通知   我的 │
└──────────────────────────────┘
```

主卡片选择规则：

```text
1. 如果存在未完成 ReviewSession，展示最近复习的未完成章节。
2. 如果没有未完成 ReviewSession，但存在已生成未复习章节，展示最近生成的可开始章节。
3. 首页只展示一个章节卡片。
4. 其他章节从“章节”页进入。
```

跳转：

```text
[继续复习] -> 恢复未完成 ReviewSession
[开始复习] -> 章节详情页或直接创建 ReviewSession 后进入题卡页
底部 [章节] -> 章节页
底部 [+] -> 添加知识页
底部 [通知] -> 通知页
底部 [我的] -> 我的页
```

#### 9.3.3 提交成功浮窗

添加新内容或从失败页点击“重新生成”提交成功后，立即回到首页，并在首页背景上展示提交成功浮窗。该浮窗只表示提交成功，不表示处理进度。

```text
┌──────────────────────────────┐
│ 复习                         │
├──────────────────────────────┤
│                              │
│      ┌────────────────┐      │
│      │ 已提交，正在生成 │      │
│      │ 完成后会通知你   │      │
│      │                │      │
│      │    [知道了]     │      │
│      └────────────────┘      │
│                              │
├──────────────────────────────┤
│  首页      章节      + 添加  通知   我的 │
└──────────────────────────────┘
```

跳转：

```text
[知道了] -> 关闭浮窗，回到首页可复习状态或首页空状态
底部 [章节] -> 查看处理中章节
底部 [通知] -> 查看生成状态消息
```

关闭浮窗后，首页不显示“正在生成”。处理中状态只在章节页显示；生成成功或失败通过通知页和系统通知告知。

### 9.6 通知流程

通知页只用于承载生成成功、生成失败等需要用户回到章节处理的消息。它不是复杂消息中心，而是任务型通知箱：通知负责把用户带回章节；任务处理后通知自动归档，同时提供轻量手动移除能力。

通知生命周期规则：

```text
点击成功通知 -> 进入章节详情页 -> 通知标记已读并自动归档，从通知页消失
点击失败通知 -> 进入失败章节详情页 -> 通知标记已读但保留，直到用户处理或手动移除
失败页点击“不再提示” -> 对应失败通知从通知页移除
手动重新生成成功 -> 对应失败通知从通知页移除，生成成功通知可出现
删除章节 -> 该章节相关通知一并移除
左滑“移除” -> 单条通知从通知页归档，不删除章节
清空已读 -> 已读通知从通知页归档，未读失败通知保留
```

通知页可以用轻量视觉状态区分未读和已读，例如未读标题加粗或显示小点；不提供复杂批量选择、搜索、归档箱或消息分类。

### 9.14 核心用户流程

#### 9.14.1 新用户成功流程

```text
空状态首页
  -> 点击底部 +
  -> 添加知识
  -> 首次添加通知权限说明
  -> 首页提交成功浮窗：已提交，正在生成
  -> 关闭浮窗后首页恢复正常

  -> 开始复习
  -> 题卡页
  -> 解释页
  -> 章节总结页
```

#### 9.14.2 生成失败流程

```text
添加知识
  -> 首页提交成功浮窗：已提交，正在生成
  -> 关闭浮窗后首页恢复正常
  -> 如果首次题目质量检查 0 道通过，系统自动重新生成一次，不通知用户
  -> 通知：生成失败
  -> 章节详情页（生成失败）
  -> 重新生成 / 查看全部知识点 / 不再提示
```

#### 9.14.3 复习强化流程

```text
题卡页
  -> 答错或不知道
  -> 解释页
  -> 继续复习
  -> 强化队列延迟出现
  -> 直到该知识点本轮答对
  -> 章节总结页
```

#### 9.14.4 中断后恢复流程

```text
题卡页未作答退出
  -> 首页主卡片显示继续复习
  -> 回到同一道题

解释页未点继续退出
  -> 首页主卡片显示继续复习
  -> 回到该解释页

章节总结页退出
  -> ReviewSession 已完成
  -> 回章节详情页
```

#### 9.14.5 题目反馈流程

```text
解释页
  -> 题目有问题
  -> 选择反馈原因
  -> 严重反馈：撤销分数变化，题目退出复习池
  -> 太简单：不撤销分数，对当前用户降权
  -> 继续复习
```

## 10. Data Model

MVP 中不单独拆分 `Source` 对象。因为当前规则是“一次有效添加默认创建一个章节”，所以 `Chapter` 同时承担原始来源记录和学习章节的职责。

后续如果支持“多个来源合并为一个主题章节”，再拆分 `Source` 和 `Chapter`。

### 10.1 Chapter

表示用户一次添加后生成的学习章节，同时保存原始来源信息。

字段包括：

- `id`
- `title`
- `source_type`: text, article_link, video_link
- `source_url`
- `source_account_or_platform`
- `raw_input`
- `extracted_text`
- `status`
- `processing_status`
- `knowledge_point_count`
- `question_count`
- `created_at`
- `updated_at`
- `dismissed_from_notifications`: 用户是否关闭通知页提示

### 10.2 KnowledgePoint

表示从章节中提炼出的一个可复习知识点。

字段包括：

- `id`
- `chapter_id`
- `title`
- `summary`
- `knowledge_type`: concept, judgment, method, scenario, misconception, comparison, step
- `source_snippet`
- `mastery_score`: 0-100，初始值 50
- `answered_count`
- `last_reviewed_at`
- `last_decay_applied_at`
- `created_at`
- `updated_at`

`new` 不需要单独存字段。判断规则是：`answered_count = 0`。

### 10.3 Question

表示一个用于测试知识点的题目。

字段包括：

- `id`
- `chapter_id`
- `knowledge_point_id`
- `question_type`: multiple_choice, true_false, scenario_judgment
- `stem`
- `options`
- `correct_answer`
- `explanation`
- `correct_understanding`
- `common_misconception`
- `source_snippet`
- `quality_score`
- `quality_status`: passed, rewrite_needed, discarded
- `created_at`
- `updated_at`

### 10.4 ReviewSession

表示用户对某个章节的一轮复习。

同一个章节同时只允许存在一个未完成 ReviewSession。完成后，该 ReviewSession 关闭；MVP 的章节总结页不提供“再来一轮”按钮。

字段包括：

- `id`
- `chapter_id`
- `status`: active, completed, abandoned
- `queue`
- `reinforcement_queue`
- `current_index`
- `started_at`
- `completed_at`
- `created_at`
- `updated_at`

`queue` 保存本轮计划复习的知识点和题目顺序。`reinforcement_queue` 保存答错或点击“不知道”后需要再次出现的知识点。

### 10.5 ReviewAttempt

表示用户每次答题行为。

字段包括：

- `id`
- `review_session_id`
- `chapter_id`
- `knowledge_point_id`
- `question_id`
- `answer`
- `result`: correct, wrong, unknown
- `is_reinforcement`
- `mastery_score_before`
- `mastery_score_after`
- `invalidated_by_feedback`
- `skipped_due_to_question_feedback`
- `answered_at`

### 10.6 QuestionFeedback

表示用户对题目质量的反馈。

字段包括：

- `id`
- `question_id`
- `knowledge_point_id`
- `chapter_id`
- `review_session_id`
- `feedback_type`: answer_wrong, too_easy, unclear, unrelated_to_source
- `severity`: severe, light
- `action_taken`: removed_from_pool, downranked_for_user, skipped_for_session
- `created_at`

题目收到严重反馈后，应退出默认复习队列，但不自动更新知识点 `mastery_score`。

如果严重反馈发生在已经更新 `mastery_score` 之后，系统需要通过对应 `ReviewAttempt.mastery_score_before` 撤销本次分数变化。`too_easy` 属于轻反馈，不撤销分数，只对当前用户降权。

## 11. Future Expansion Notes

### 11.1 Split Source and Chapter

MVP 中 `Chapter` 直接保存来源信息。未来如果支持以下能力，需要拆分 `Source` 和 `Chapter`：

- 多篇文章合并成一个主题章节
- 多个视频合并成一个课程式章节
- 一篇来源拆成多个章节
- 同一来源被多个学习目标复用
- AI 建议合并同主题章节

未来结构可以变成：

```text
Source
  -> 原始文章、视频、文字

Chapter
  -> 一个学习单元
  -> 可以关联一个或多个 Source
```

### 11.2 More Advanced Memory Model

MVP 使用简单 `mastery_score` 和每日 `-3` 衰减。未来可以扩展为更动态的记忆模型，例如：

- 根据题型难度调整分数变化
- 根据用户反应时间调整掌握度
- 根据连续答对次数降低复习频率
- 根据知识点类型使用不同遗忘速度
- 根据用户历史表现个性化衰减曲线

## 12. Technical Considerations

- iOS 客户端可以先使用原型或 SwiftUI 实现核心体验。
- MVP 需要账号系统，因为用户上传内容、生成结果和复习记录需要存储在服务器端，并绑定到具体用户。
- 账号系统第一版只承担登录、退出登录、用户数据归属和云端恢复，不承担订阅、会员、支付或社交关系。
- 文章链接正文提取可能受平台限制，失败时必须提供清晰失败原因和重试入口。
- 视频链接处理依赖字幕、转写文本、平台摘要或用户粘贴文本。MVP 不要求直接理解视频画面。
- AI 生成流程需要拆成内容解析、知识点提炼、题目生成、质量检查、复习调度五个阶段。
- 复习调度第一版以知识点为核心对象，题目是知识点的测试载体。
- 题目质量需要结构化评分或规则判断，至少覆盖来源可支撑性、答案唯一性、重复度和理解深度。
- iOS 通知权限应在用户第一次添加成功后请求，不在 App 首次启动时请求。

## 13. Privacy and Data Handling

MVP 必须清楚传达用户上传内容的使用边界。

产品要求：

- 用户必须登录后才能把内容保存到云端并生成复习内容。
- 用户上传的文字、文章链接、视频链接和提取文本仅用于生成章节、知识点、题目、解释和复习记录。
- 用户数据必须绑定到用户账号，不得被其他用户访问。
- 用户内容不公开、不共享、不进入公共题库。
- 删除章节会永久删除该章节关联的原始内容、提取文本、知识点、题目和复习记录。
- 题目质量反馈仅用于改进该用户的复习体验和后续生成质量。

删除确认弹窗必须明确删除影响，避免用户误删。

## 14. MVP Acceptance Flow

MVP 的端到端验收路径如下：

```text
1. 新用户打开 App，看到空状态提示和底部中间 `+` 添加入口。
2. 用户粘贴一篇文章链接。
3. 用户第一次添加成功后，App 解释通知用途并请求 iOS 通知权限。
4. App 返回首页，并展示“已提交，正在生成，完成后会通知你”的提交成功浮窗；用户关闭浮窗后首页不显示处理中状态，详细处理阶段只在章节页或通知页查看。
5. 章节生成成功后，系统发送通知。
6. 用户点击通知，进入章节详情页。
7. 章节详情页展示标题、来源、链接、知识点数、题目数、状态和知识点列表。
8. 用户点击“开始复习”。
9. 系统创建 ReviewSession，生成章节复习队列。
10. 用户答题；答错或点击“不知道”进入统一解释页。
11. 答错或“不知道”的知识点进入强化队列，直到本轮答对。
12. 用户可以在解释页反馈“题目有问题”，系统按反馈规则撤销分数或降权。
13. 用户中途退出后，再次进入可继续未完成 ReviewSession。
14. 本章所有知识点都在本轮答对后，进入章节总结页。
15. 如果存在下一章，用户可以点击“继续下一章”；否则可以回到章节。
16. MVP 上线前，题目生成链路必须通过固定测试集评估。
```

失败验收路径：

```text
1. 用户粘贴文章链接或视频链接。
2. 后台处理失败，或题目第一次质量检查 0 道通过。
3. 如果是题目质量检查 0 道通过，系统自动重新生成一次，不通知用户。
4. 如果第二次仍然 0 道题通过，或其他处理阶段失败，系统发送生成失败通知。
5. 用户点击通知进入章节详情页的失败状态。
6. 页面展示用户可理解的失败原因和可操作按钮。
7. 如果已提取知识点，用户可以重新生成题目、通过“查看全部 X 个”查看知识点，或选择“不再提示”。
8. 来源作为页面信息中的可点击链接展示；复制文字来源显示“原文：查看输入内容”。
9. 被关闭的失败章节不再出现在通知页中，但仍保留在章节页。
```

## 15. Success Metrics

MVP 最核心指标是题目质量。

建议用以下指标判断：

- 生成题目可用率：通过系统质量检查并进入复习池的题目比例。
- 用户低质量反馈率：被用户标记为“不合适”的题目比例。
- 来源支撑率：抽样检查中，题目答案能被原始来源支撑的比例。
- 理解型题目占比：场景判断题和概念辨析类选择题在全部题目中的比例。
- 单次复习完成率：用户开始一次章节复习后完成该组题目的比例。
- 次日复习回访率：用户第二天是否再次打开并完成复习。
- 测试集可用题比例：固定测试集中人工认为可以直接进入复习的题目比例。
- 测试集严重问题比例：固定测试集中来源不支撑、答案不唯一、解释错误等严重问题的比例。

## 16. Open Questions

1. 第一版 iOS 是做真实可安装 App，还是先做高保真可点击原型？
2. 题目质量测试集第一版选哪些样本？每类内容各选多少条？
3. 题目质量测试集的人工评分执行表如何设计？例如来源支撑、答案唯一、理解深度分别如何打分？

7. 是否需要允许用户手动编辑知识点或题目，还是第一版只支持删除和反馈？

## 17. MVP Scope Closure

本节用于进入开发计划前做范围收口，避免实现阶段把后续想法混进 MVP。

### 17.1 MVP 必须包含

- 账号登录：用户必须登录后才能添加内容、生成题目和保存复习记录。
- 添加内容：支持粘贴文字、文章链接、视频链接。
- 异步生成：提交后回到首页并展示提交成功浮窗，处理状态进入章节页和通知页。
- 章节系统：一次有效添加默认生成一个章节，章节保存来源信息、知识点、题目和处理状态。
- AI 生成链路：内容清洗、语义分块、知识点提取、题目生成、质量检查、入池。
- 题目质量：上线前必须使用固定测试集评估题目质量。
- 复习流程：首页开始/继续复习，章节详情开始复习，题卡答题，解释页，章节总结页。
- 复习调度：按知识点维护掌握状态；答错或“不知道”的知识点本轮继续出现，直到答对。
- 题目反馈：用户可以反馈题目问题，反馈后的题目不再进入默认复习队列。
- 来源回看：题目和章节都能回到来源信息；链接来源显示可点击链接，文字来源进入输入内容页。
- 完整知识点页：只用于查看本章全部知识点，不承担编辑、筛选、收藏或调度。
- 通知：只承载生成成功和生成失败，不做复杂消息中心。

### 17.2 MVP 明确不做

- 截图上传、OCR、小红书截图识别。
- 用户入池前逐题确认。
- 首页推荐系统、每日强制学习计划、打卡体系。
- 单独薄弱点页、错题本页、知识点掌握度看板。
- 章节搜索、筛选、标签、文件夹、收藏、归档、批量管理。
- 通知左滑已读、左滑删除、批量已读、通知归档。
- 订阅、会员、套餐、付费墙、Free Plan 展示。
- 公共题库、内容社区、团队协作。
- 用户手动编辑知识点或题目。
- AI 自动合并同主题章节。

### 17.3 进入开发计划前需要技术预研

- 公众号文章和普通网页正文提取：确认第一版可用方案、失败率和备用策略。
- 视频链接文本提取：确认字幕、转写、简介或平台文本的获取方式。
- 账号方案：确认 Apple 登录、手机号登录或其他登录方式的 MVP 优先级。
- 异步生成链路：确认后台任务、重试、状态同步和通知发送方案。
- 题目质量测试集：确认样本类型、评分表和上线门槛。

### 17.4 正式开发前的真实体验验证版 Demo

在正式进入完整 iOS MVP 开发前，建议先做一个轻量真实体验 Demo，用于验证产品核心价值，而不是只验证页面点击。

Demo 目标：

- 验证用户添加真实内容后，AI 是否能生成有价值的知识点和题目。
- 验证用户从生成结果进入做题后，是否觉得复习节奏自然、有用。
- 验证题目解释、来源片段和题目反馈是否能建立信任。

Demo 范围：

- 可以使用 HTML 或轻量 Web 前端，不要求真实 iOS App。
- 当前 Demo 支持用户粘贴真实文字内容和普通文章链接。
- 文章链接输入层位于 generation-engine 之前；链接先提取标题、来源、正文和原链接，再复用同一套出题引擎。
- 当前 Demo 对视频链接只做识别和友好失败提示，暂不提取视频文本。
- 后端真实调用 AI 生成知识点和题目。
- 展示生成中、章节详情、题卡、解释页和章节总结。
- 不要求账号、真实推送通知、完整章节管理、掌握度长期存储。

Demo 验收：

- 用户能粘贴一段真实内容并成功生成章节。
- 用户能粘贴普通网页文章链接并生成章节；如果正文提取失败，看到“文章正文提取失败”的可理解提示。
- 用户粘贴视频链接时，不进入出题链路，看到当前 Demo 暂未接入视频文本提取的提示。
- 系统能生成知识点和可作答题目。
- 用户能完成至少一轮真实做题。
- 测试用户能评价题目是否有用、解释是否清楚、复习节奏是否舒服。
- Demo 结果用于调整 AI 生成逻辑和复习流程，再进入完整 iOS MVP 开发。

### 17.5 当前 PRD 定稿口径

- 产品名：拾贝。
- Slogan：每天捡起一枚知识贝壳。
- 底部导航：`首页 / 章节 / + 添加 / 通知 / 我的`。
- 首页只承担开始或继续复习，不展示章节管理、失败处理或复杂推荐。
- 添加成功后返回首页并展示提交成功浮窗；关闭后首页不显示处理中状态。
- 生成成功或失败通过通知页和系统通知告知用户。
- 失败章节允许进入章节详情页查看原因、来源和已提取知识点。
- 如果失败章节已提取知识点，页面直接展示部分知识点，并通过“查看全部 X 个”进入完整知识点页；按钮区不再放“查看知识点”按钮。
- 来源入口以链接形式出现在来源信息区，不作为操作按钮。
- 账号是 MVP 必需能力；订阅和会员不是 MVP。
