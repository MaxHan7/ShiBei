# 拾贝知识点逻辑梳理与迭代方向

## 1. 这份文档的目的

知识点是拾贝出题系统的第一层基础。题目、解释、复习调度和章节总结都依赖知识点质量。如果知识点识别错了，后面的题目即使格式正确，也会偏离用户真正想复习的内容。

这份文档先记录当前后端知识点逻辑，再整理下一版“主线 + 可用方法型”知识点设计方向。当前共识是：MVP 阶段不硬编码固定知识点数量，而是按文章长短和内容密度动态决定；整体保持精简入池，以文章主线为骨架，以可迁移的方法、判断原则和适用场景为主要复习对象，少量保留关键误区和边界。

## 2. 当前旧逻辑

### 2.1 整体链路

当前生成链路大致是：

```text
原始内容
-> 正文提取 / 清洗
-> 语义分块
-> 标记正文块和导读块
-> 模型提取知识点候选
-> 确定性过滤
-> 根据来源位置排序
-> 基于知识点生成题目
-> 质量检查和题目入池
```

知识点不是由前端生成，而是后端服务端生成。iOS 和 HTML Demo 只负责展示知识点、题目和复习状态。

### 2.2 正文清洗与分块

后端先得到 `cleanedText`，再按段落切分。

当前分块规则：

- 短段落直接成为一个 chunk。
- 长段落按句子拆分，尽量控制在数百字以内。
- 每个 chunk 会获得一个粗略类型。

当前 chunk 类型主要包括：

- `concept`
- `judgment`
- `method`
- `scenario`
- `counterexample`
- `comparison`
- `step`

这些类型现在主要通过关键词判断，例如“步骤、第一、然后”更容易被识别成 `step`，“区别、相比、而不是”更容易被识别成 `comparison`。

### 2.3 导读 / 金句 / 开头摘要处理

当前系统已经增加了 `lead_summary` 识别。

文章开头的导读、金句摘录、编辑摘要、短句核心价值总结会被标记为 `lead_summary`。这类内容可以帮助模型理解文章主题，但不能作为知识点的 `sourceQuote`。

当前规则是：

- 如果某个观点只出现在 `lead_summary`，不能成为知识点。
- 如果某个观点先出现在 `lead_summary`，但正文后面有展开解释，必须锚定正文段落。
- 如果候选知识点的 `sourceQuote` 只来自导读区，会被过滤为 `lead_summary_source`。

这个逻辑是为了避免系统从文章开头的“核心句摘录”里直接出题，因为那通常不是原文论证位置，只是编辑摘要或作者金句。

### 2.4 模型提取知识点候选

模型收到带有 chunk 标记的正文，返回知识点候选。

每个候选当前包含：

- `title`：知识点标题。
- `knowledgeType`：知识点类型。
- `summary`：简短总结。
- `keyClaim`：核心判断。
- `sourceQuote`：支持该知识点的原文片段。
- `testabilityReason`：为什么值得复习 / 出题。
- `questionAngles`：适合从哪些角度出题。
- `testabilityScore`：可考性评分。

当前 prompt 的核心要求是：

- 知识点不能只是原文摘抄。
- 每个知识点必须有来自输入文本的 `sourceQuote`。
- 优先保留概念边界、方法原则、适用场景、常见误区、反例、对比、步骤。
- 过滤纯情绪表达、孤立句、常识、碎片细节和来源无法支撑的推断。
- 数量根据内容密度动态决定，不套固定数量。
- 清单、步骤、案例、多机会点结构要覆盖主要结构。

### 2.5 确定性过滤

模型返回候选后，后端会做确定性过滤。

当前过滤原因包括：

- `incomplete_fields`：标题、总结、核心判断等字段不完整。
- `source_not_supported`：`sourceQuote` 无法在原文中匹配。
- `lead_summary_source`：来源只来自导读 / 金句 / 开头摘要。
- `low_testability`：可考性评分低于阈值。
- `fragmented_information`：信息太碎，不像完整知识点。
- `emotional_expression`：纯情绪表达。
- `common_sense`：过度宽泛的常识。
- `duplicate`：和已保留知识点重复。

保留下来的进入 `knowledgePoints`，被过滤的进入 `filteredKnowledgePoints`，用于调试和质量分析。

### 2.6 来源定位与排序

保留下来的知识点会根据 `sourceQuote` 在 `cleanedText` 中的位置排序。

当前定位方式：

- 先做原文 exact match。
- 再做去空白、去标点后的 normalized match。
- 最后用 `sourceQuote` 前缀做模糊匹配。

定位结果写入：

- `sourceOrder`
- `sourceStartOffset`
- `sourceEndOffset`

后续题目也会按对应知识点的 `sourceOrder` 排序，因此用户复习时大体会沿着文章先后顺序推进。

## 3. 当前逻辑的优点

当前逻辑已经有几个重要基础：

1. 知识点有来源锚点，不是凭空摘要。
2. 导读 / 金句不能直接作为知识点来源。
3. 知识点会按原文位置排序。
4. 出题以知识点为中心，而不是全文随机出题。
5. 被过滤的知识点会保留诊断信息，方便后续分析。

这些方向都应该保留。

## 4. 当前逻辑的主要问题

### 4.1 知识点更像“可出题片段”，还不够像“文章理解地图”

当前系统主要问的是：这个候选能不能复习、能不能出题、来源是否支撑。

但它还没有充分判断：这个知识点在文章整体理解里承担什么角色。

结果是，有些局部例子、局部判断、好出题的句子，可能和文章主干观点获得接近的权重。

### 4.2 缺少文章结构角色

当前只有 `knowledgeType`，例如 `method`、`comparison`、`scenario`。但这描述的是知识点形态，不描述它在文章里的结构地位。

下一版需要区分：

- 核心论点
- 关键论据
- 方法步骤
- 边界 / 反例 / 误区
- 重要案例
- 背景信息
- 低价值细节

同样是 `scenario`，有的场景是文章主线的一部分，有的只是插图式例子。当前系统不容易分清。

### 4.3 缺少覆盖目标

当前 prompt 说“动态决定数量”，但没有明确“复习完这些知识点后，用户应该能重建文章主线”。

这会带来两个问题：

- 可能漏掉文章主线中的必要节点。
- 可能保留一些好出题但不关键的点。

### 4.4 知识点质量评估维度不足

现在过滤维度偏工程正确性：字段、来源、重复、可考性、碎片。

但知识点本身还应该评估：

- 是否命中文章核心。
- 是否代表一个必要理解节点。
- 是否只是局部例子。
- 是否过抽象。
- 是否过碎。
- 是否和其他知识点重叠。
- 是否有足够上下文支撑后续出题。

### 4.5 `sourceQuote` 偏短时容易带偏题目

题目的 `sourceSnippet` 已经升级成更完整的原文上下文，但知识点的 `sourceQuote` 仍常常是一句短证据。

如果 `sourceQuote` 太短，模型可能围绕短句出题，而不是围绕该段落真正表达的上下文出题。

## 5. 新逻辑：主线 + 可用方法型知识点

### 5.1 产品定义

下一版知识点应定义为：

> 用户复习完这一组知识点后，应该能重建这篇文章的核心观点、论证路径和关键判断。

知识点不是“文章里所有可考信息点”，而是“文章理解地图中的关键节点”。

MVP 阶段采用“主线 + 可用方法型”结构：

> 以文章主线为骨架，以可迁移的方法、判断原则和适用场景为主要复习对象，少量保留关键误区和边界。

它比纯主线骨架更有学习价值，也比深度误区边界型更适合 MVP。用户复习完不仅知道文章讲了什么，还能带走几个以后可能用得上的判断和方法。

### 5.2 数量目标

第一阶段采用动态但克制的数量：

- 短内容通常保留 1-3 个知识点。
- 普通文章通常保留 4-8 个知识点。
- 长文、清单、访谈或多段论证文章可以保留 8-12 个，必要时更多。
- 不追求把所有可出题片段都保留。

这些区间是产品倾向，不是硬指标。系统不为凑数保留低价值细节，也不为压缩而砍掉必要主线节点。核心原则是：足够重建文章主线，同时保持精简。

MVP 推荐比例：

- `main_claim`：约 30-40%，覆盖文章核心观点。
- `supporting_reason` / `method_step`：约 40-50%，保留可迁移的方法、原则、理由和适用场景。
- `boundary`：约 10-20%，保留关键误区、边界或反例。
- `case_evidence`：只在案例本身承载核心论证时保留。
- `background` / `detail`：默认不入池。

### 5.3 知识点分层

下一版建议引入知识点结构角色，暂定为：

#### 主干知识点

文章真正想让读者带走的核心判断。

特点：

- 能概括文章的一段关键论证。
- 不依赖太多局部细节也能成立。
- 适合出理解题、判断题、场景题。

#### 支撑知识点

支持主干观点的理由、机制、对比或关键论据。

特点：

- 不是文章最终结论，但解释了为什么结论成立。
- 常常回答“为什么”“如何发生”“区别在哪里”。

#### 方法 / 原则知识点

用户可以迁移到工作、学习或判断中的方法、原则、操作路径和适用场景。

特点：

- 不只是文章局部信息，而是可被用户以后调用的判断工具。
- 常常回答“应该怎么做”“什么情况下适用”“怎样判断”。
- 是 MVP 阶段最应该优先保留的知识点类型之一。

#### 边界知识点

常见误区、反例、适用条件、不适用条件。

特点：

- 帮助用户避免误解。
- 非常适合做场景判断题。

#### 低价值信息

不应默认进入复习池。

包括：

- 金句和开头摘要。
- 情绪表达。
- 单点事实。
- 只服务文风的例子。
- 没有迁移价值的细节。
- 只适合收藏但不值得复习的句子。

### 5.4 新的选择标准

一个知识点进入 `knowledgePoints` 前，应同时满足：

1. 来源可追溯：能在正文中找到支撑段落。
2. 主线相关：对重建文章核心理解有帮助。
3. 表述完整：不是孤立短句或标题党判断。
4. 可复习：能被题目检验，而不是只能被背诵。
5. 不重复：和其他知识点不是同一观点的不同说法。
6. 不过碎：记住它后用户真的获得理解增量。

### 5.5 新的输出字段方向

后续可考虑在知识点候选中新增或派生这些字段：

- `structureRole`：`main_claim` / `supporting_reason` / `method_step` / `boundary` / `case_evidence` / `background` / `detail`
- `importanceScore`：文章主线重要度。
- `coverageReason`：为什么这个点应该进入复习池。
- `parentClaimId`：如果是支撑点，挂到哪个主干观点下。
- `sourceContext`：比 `sourceQuote` 更完整的支撑上下文，可用于后续题目和解释。

第一版不一定要全部落地，但 `structureRole` 和 `importanceScore` 最值得优先做。

## 6. 建议的新生成流程

下一版可以从现在的一步式候选提取，升级为两层逻辑：

```text
正文
-> 清洗和分块
-> 识别文章结构 / 主线
-> 提取知识点候选
-> 给候选标注结构角色和重要度
-> 按“主线 + 可用方法型”策略动态选择知识点
-> 来源定位和上下文扩展
-> 基于知识点出题
```

核心变化不是多一次模型调用本身，而是让系统先理解“文章在讲什么、怎么论证”，再决定“哪些点值得复习”。

## 7. 质量评估标准

知识点本身也应该进入质量工作台评估，而不只评估题目。

建议人工 / AI 预标注时增加知识点维度：

- `mainline_relevance`：是否属于文章主线。
- `source_support`：来源是否支撑该知识点。
- `granularity`：是否太碎或太泛。
- `uniqueness`：是否和其他知识点重复。
- `review_value`：复习它是否有价值。
- `coverage_role`：主干 / 支撑 / 边界 / 案例 / 背景 / 细节。
- `missing_core_point`：是否漏掉明显核心观点。

这些维度可以帮助判断问题到底出在：

- 知识点提取阶段。
- 知识点筛选阶段。
- 题目生成阶段。
- 来源上下文选择阶段。

## 8. 下一步实施建议

建议下一步不要马上大改出题 prompt，而是先做一个小步迭代：

1. 更新知识点提取 prompt，明确“主线 + 可用方法型”和动态精简目标。
2. 增加 `structureRole` 和 `importanceScore`。
3. 增加候选选择规则：优先主干，其次可迁移的方法 / 原则 / 支撑理由，再保留少量关键边界，减少局部细节。
4. 在质量工作台中展示每个知识点的结构角色和重要度。
5. 用现有测试集跑 5-10 篇文章，人工检查知识点是否能重建文章主线。

第一轮成功标准：

- 知识点数量更克制。
- 每个知识点都能说清“为什么值得复习”。
- 文章主线覆盖更完整。
- 低价值细节减少。
- 题目来源和解释不再被局部短句带偏。

## 9. 和题目可信度闭环的关系

知识点逻辑决定“值得复习什么”，题目可信度闭环决定“这道题是否可信”。两者需要分开诊断：

- 如果知识点偏题、过碎或不属于文章主线，问题归因到知识点提取 / 筛选。
- 如果知识点正确，但题目答案、解释或干扰项不可靠，问题归因到题目生成。
- 如果题目方向正确，但来源片段不能帮助用户回顾原文，问题归因到来源上下文选择。

题目层新增的可信度诊断包括：

- `answerGroundingScore`：正确答案是否被来源上下文支撑。
- `explanationFaithfulnessScore`：解释是否忠实于来源和正确答案。
- `contextRelevanceScore`：来源片段是否能解释题目，而不是只证明某句话出现过。
- `misconceptionSupportScore`：常见误区是否与来源和题目相关。

低置信题不代表知识点一定错误。它表示题目仍可复习，但需要在质量工作台中优先人工检查原因，例如来源弱、解释弱、上下文弱、题型偏差或 judge 建议重写。第一版用户侧不展示低置信标签，避免打断复习体验。
