# AI 知识复习产品设计稿

日期：2026-04-28

## 产品定位

这个产品可以理解为一个“个人知识版百词斩”：用户把日常随机收集到的知识丢进去，AI 自动把它们整理成可以复习、可以测试、可以反复巩固的题目。

核心承诺是：

> 用户只负责投喂内容和刷题，知识提炼、分类、出题、解释、复习安排都交给 agent。

第一版不应该先做成传统知识库。它的核心体验应该是复习优先：用户打开 App 后，不是先整理资料，而是直接进入“今日复习”，用几分钟把自己曾经收藏过的碎片知识重新唤醒。

## 目标用户

第一批用户是经常收集碎片知识的人。

他们可能会从这些地方获得内容：

- 公众号文章
- 小红书笔记
- 产品理论文章
- AI 和大模型知识
- 商业分析、增长、运营、管理方法论
- 自己随手复制或截图的学习材料

这类用户通常不是不想学习，而是不想花大量时间整理。他们真正需要的是一种低成本的机制，把“我觉得有用，所以先存一下”的内容变成之后真的能记住、能用起来的知识。

## 核心使用场景

1. 用户在日常浏览中看到一段有价值的内容。
2. 用户把文字、截图或链接像发消息一样丢进产品。
3. agent 自动识别内容，提炼概念、判断、场景和反例。
4. 系统生成带解释和来源的测试题。
5. 用户之后打开 App，直接进入“今日复习”。
6. 系统根据用户答题结果安排之后的复习节奏。

这个产品的重点不是“帮用户收藏更多东西”，而是“让用户收藏过的东西真的进入记忆”。

## 产品原则

- 复习是主产品，知识管理只是后台能力。
- 投喂内容必须像发消息一样简单。
- 题目要测试理解，而不是只测试字面记忆。
- 每道题都应该有简短解释和来源追溯。
- 第一版避免做复杂知识图谱、公开题库、多人协作、深度导出等重功能。
- 整体体验要轻、快、有进度感，接近背单词产品的习惯养成模式。

## MVP 范围

### 输入方式

第一版支持三种输入：

- 粘贴文字
- 上传截图或图片
- 粘贴链接

暂时不需要打通公众号、小红书、浏览器插件或平台 API。第一版只要有一个聊天式投喂入口，就足够验证核心体验。

### 内容范围

第一版优先支持中文内容，重点领域包括：

- 产品经理方法论
- AI 和大模型知识
- 商业框架
- 文章观点
- 学习笔记

这些内容天然适合被提炼成概念、判断、适用场景和反例。

### 复习体验

默认首页是“今日复习”。

用户每天看到一个小的复习队列，比如 10 到 15 道题。每道题应该能在 10 到 20 秒内完成，整体复习时间控制在 3 分钟左右。

答题后展示：

- 对错反馈
- 简短解释
- 对应知识点
- 来源引用

核心目标是让用户形成习惯：打开、刷几题、知道自己哪里懂了哪里没懂，然后离开。

### 题型

第一版优先支持这些题型：

- 场景判断题
- 选择题
- 判断题
- 概念辨析题

其中“场景判断题”应该优先级最高。因为产品理论、AI 知识、商业方法论这类内容，如果只背定义很容易浅；真正有价值的是判断“这个理论适合什么场景，不适合什么场景，容易被误用在哪里”。

### 管理功能

第一版只做轻量管理：

- 主题分组
- 薄弱点列表
- 错题复习
- 来源回看

不做复杂标签体系、知识图谱、公开分享、团队协作或重导出功能。

## 主要界面

### 今日复习

这是默认首页。

页面展示今日题目数量、当前进度和一道题卡片。用户快速作答后，系统立刻展示解释，然后进入下一题。

这个页面的设计目标是建立日常习惯，而不是展示信息量。它应该像一个轻量训练场，而不是一个资料库。

### 投喂页

投喂页像一个轻量聊天窗口。

用户可以发送：

- 文字
- 截图
- 链接

每条内容都有处理状态：

- 识别中
- 已生成题目
- 需要确认
- 处理失败

用户不需要在投喂前手动分类。agent 应该自动推断主题，只在确实不清楚的时候再询问用户。

### 薄弱点

这个页面展示用户最近容易错、记不牢或反复模糊的知识点。

它不是完整知识库，而是一个“该补哪里”的清单。用户可以从这里进入专项强化复习。

### 来源

来源页保留用户原始投喂的内容。

用户可以从一道题回到原始片段、截图或链接。这个能力很重要，因为它能增加用户对 AI 出题的信任，也方便用户在需要时做深度回看。

## Agent 处理流程

### 1. 内容解析

agent 根据输入类型做不同处理：

- 截图和图片先做 OCR。
- 链接尽量提取正文内容。
- 粘贴文字直接切分和清洗。

系统记录来源标题、来源类型、创建时间和原始内容。

### 2. 知识点提炼

agent 从内容中提炼三类知识：

- 核心概念
- 关键判断
- 适用场景和反例

例如，一段关于 RAG 的内容可以被提炼为：

> RAG 适合解决知识更新和事实检索问题，但它不等于让模型拥有真正的长期记忆。

这种表达比单纯摘一句原文更适合后续出题。

### 3. 题目生成

每个可复习知识点应尽量至少生成 1 道可复习题；高价值知识点理想状态生成 1 到 3 道题，用不同题型和不同考察角度帮助用户反复激活同一个关键理解。进入复习池的题目都应被 ReviewSession 消费，而不是只抽每个知识点的一道代表题。

每道题包含：

- 题干
- 选项
- 正确答案
- 答案解释
- 来源引用
- 对应知识点
- 难度或质量评分

题目不追求堆数量，而追求覆盖主要知识点并能检验理解。题型由系统推荐，但推荐题型不是硬性失败条件；如果另一种题型更自然、更能考理解，也可以进入复习池。

### 4. 质量控制

agent 需要过滤、重写或低置信保留这些题目：

- 太简单，只考原文字面记忆
- 答案模糊，存在多个合理选项
- 无法从来源内容中支撑
- 依赖来源之外的未经确认假设
- 和已有题目高度重复

质量检查主要用于排序、重写和低置信标记。结构完整、来源可支撑、答案唯一但质量不够高的题可以作为低置信题进入复习池；缺正确答案、选项结构错误、来源完全不支撑或答案不唯一的题不能入池。只有最终 0 道可复习题时，章节才算题目生成失败。

### 5. 复习调度

系统为每道题维护本轮复习状态，并把题目结果聚合成知识点掌握状态：

- 新知识
- 已熟悉
- 模糊
- 答错
- 反复答错
- 已掌握

章节内首次复习应按文章实际内容先后推进。系统为知识点和题目保存 `sourceOrder` 或原文 offset，首次创建复习队列时按这个顺序排列，让用户像重新走一遍文章主线一样主动回忆；答错、选择“不知道”或后续长期复习时，再按强化规则和掌握度调整顺序。

ReviewSession v2 采用按题目驱动的状态机：一轮复习的事实来源是题目队列，所有入池题都会进入队列；同一知识点的多道题用于理解、辨析、应用等不同角度强化。答错或“忘记了”时，系统针对当前题目安排强化，而不是只按知识点粗略重排。为避免用户被困在同一道题上，同一题本轮最多强化 2 次；仍未答对会记录为“需回看”，但允许本轮结束。

章节完成状态是长期反馈，不等同于当前这一轮复习会话。用户完成过某章后，该章在章节列表中保持“已完成”，首页“已掌握知识点”也继续累计这部分成果；之后用户重新复习同一章，只会开启新一轮巩固，不应撤销已完成标签或把已掌握数量清零。

第一版可以使用简单的间隔复习规则：

- 答错：很快再次出现
- 模糊：明天复习
- 新题答对：几天后复习
- 多次答对：更久之后再出现

调度模型不需要一开始就很复杂，但必须可解释、可调整。

## 数据模型

第一版可以围绕四个核心对象设计。

### Source：来源

表示用户投喂的原始内容。

字段包括：

- ID
- 类型：文字、图片、链接
- 原始内容或文件引用
- 提取后的文本
- 标题
- 创建时间
- 处理状态

### Knowledge Point：知识点

表示从来源中提炼出来的一个最小知识单元。

字段包括：

- ID
- 来源 ID
- 主题
- 摘要
- 关键判断
- 适用场景或反例
- 置信度

### Question：题目

表示一个可以被复习的题目。

字段包括：

- ID
- 知识点 ID
- 题型
- 题干
- 选项
- 正确答案
- 解释
- 难度
- 质量状态

### Review Record：复习记录

表示用户每一次答题行为。

字段包括：

- ID
- 题目 ID
- 用户答案
- 是否正确
- 作答时间
- 复习时间
- 下次复习时间
- 学习状态

## 成功标准

MVP 成功的标准可以这样定义：

- 用户能在 10 秒内完成一次内容投喂。
- 系统能在无需用户手动整理的情况下生成可用题目。
- 用户能在约 3 分钟内完成 10 到 15 道题。
- 用户觉得题目确实帮助自己记住了曾经收藏过的内容。
- 用户答错或模糊的知识点会在之后自然再次出现。

## 非目标

第一版不做：

- 公开题库
- 团队协作
- 深度平台集成
- 复杂知识图谱
- 重度笔记编辑器
- 以全文总结为主的阅读工具
- 以导出为核心的题库工具

这些方向都可以以后扩展，但不应该干扰第一版的核心习惯闭环。

## 未来扩展方向

在 MVP 验证“投喂 -> 出题 -> 复习”的闭环之后，可以继续扩展：

- 手机分享入口
- 浏览器插件
- 类微信文件助手的 agent 投喂入口
- 更精细的间隔复习算法
- 个性化薄弱点报告
- 每周知识复盘
- 对话式知识教练
- 多模态内容理解
- 面向特定学习目标的知识合集

## 待确认问题

进入原型或开发计划前，需要继续确认这些问题：

- 第一版应该优先做移动端、网页端，还是聊天机器人？
- 用户是否需要编辑 AI 生成的题目？
- 题目进入复习池前，需要多少人工确认？
- 复习调度第一版应该基于“题目”还是“知识点”？
- 第一版最应该优化哪种输入：文字、截图，还是链接？
