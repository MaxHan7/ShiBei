# 拾贝 P0 生产化推进计划

> 这份文档是 `docs/production-readiness-review-zh.md` 的执行地图，用于把 P0 风险拆成可独立计划、独立实现、独立验收的阶段。第一版只定义推进顺序和验收口径，不改变公开 API、数据库 schema、iOS 契约或 Railway 运行命令。

## 当前默认决策

| 决策 | 默认方案 | 原因 |
| --- | --- | --- |
| 队列技术 | Postgres-backed job queue | 复用当前 Railway PostgreSQL，避免第一阶段引入 Redis/BullMQ 运维复杂度。 |
| worker 形态 | 独立 worker 启动命令 | API 与生成执行职责分离，便于后续独立扩容和观测。 |
| 迁移策略 | 先补初始化/迁移边界，再单独设计迁移框架 | 当前仓库没有迁移目录或迁移命令，不在总计划里直接指定完整迁移框架。 |
| 执行方式 | 按阶段单独开实施计划 | 避免一次性大重构；每个阶段完成后都要更新生产就绪度审计台账。 |

## 推进原则

- 先保住现有行为，再引入生产能力：`POST /api/chapters` 仍应快速返回 `submitted`。
- 每个阶段都必须能单独合并、单独回滚、单独验证。
- 不以“代码已写”为完成标准；必须补测试证据或运行证据。
- 每个阶段结束后，更新 `docs/production-readiness-review-zh.md` 中对应风险项的状态、证据和剩余风险。

## 阶段 0：基线确认

| 项目 | 内容 |
| --- | --- |
| 目标 | 冻结当前接口行为和风险基线，建立后续生产化改造的验证口径。 |
| 触及风险 ID | PRD-P0-001、PRD-P0-002、PRD-P0-003、PRD-P0-004、PRD-P0-005 |
| 主要改动 | 梳理当前 API 行为、生成状态流、失败态、重启恢复、测试命令和最小压测场景；不改运行逻辑。 |
| 验收标准 | 有一份阶段 0 实施记录，列出当前关键接口预期、最小压测样本、故障验证场景和后续阶段的不可破坏行为。 |
| 审计台账更新 | 保持 P0 风险为 `已识别` 或 `待验证`；补充阶段 0 产生的基线证据链接。 |

阶段 0 基线记录见 `docs/production-hardening-stage-0-baseline-zh.md`。

阶段 0 的输出应回答：

- 当前 `POST /api/chapters`、`GET /api/chapters/:id`、`POST /api/chapters/:id/regenerate` 的用户可见语义是什么。
- 当前生成任务在服务重启、模型超时、文章提取失败时如何落状态。
- 后续每个阶段必须保留哪些 iOS 端兼容行为。

## 阶段 1：生成任务队列化最小闭环

| 项目 | 内容 |
| --- | --- |
| 目标 | 把“API 接收请求”和“模型生成执行”拆开，建立 Postgres 队列表和独立 worker 最小闭环。 |
| 触及风险 ID | PRD-P0-001、PRD-P0-002 |
| 主要改动 | 新增任务队列表；API 创建章节后入队；独立 worker 领取任务并执行生成；设置 worker 并发上限；保留本地内存模式的开发回退策略。 |
| 验收标准 | API 仍快速返回 `submitted`；worker 能完成文本生成；同一时刻执行中的任务数不超过配置；worker 停止时任务不会永久处理中。 |
| 审计台账更新 | PRD-P0-001 从 `已识别` 更新为 `待验证` 或 `已缓解`，取决于是否已有并发压测和运行证据；PRD-P0-002 保持待验证，进入阶段 2。 |

阶段 1 的实施计划需要明确：

- 队列表字段、任务状态机、领取 SQL 和可见性超时。
- worker 启动命令、环境变量、并发上限和退出处理。
- poison job 处理方式：达到最大尝试次数后写入明确失败态。

阶段 1 实施计划见 `docs/production-hardening-stage-1-queue-plan-zh.md`。

## 阶段 2：幂等和多进程协调

| 项目 | 内容 |
| --- | --- |
| 目标 | 防止重复提交、重复领取、重复生成和多实例状态覆盖。 |
| 触及风险 ID | PRD-P0-001、PRD-P0-002 |
| 主要改动 | 为创建章节和重新生成设计幂等键；任务领取使用数据库级锁或条件更新；任务完成写入时校验当前任务身份；定义重复请求的用户可见响应。 |
| 验收标准 | 同一输入或同一章节重复提交不会产生不可控重复生成；两个 worker 并发时同一任务只会被一个 worker 执行；过期任务能被重新领取或明确失败。 |
| 审计台账更新 | 若双 worker 并发验证通过，将 PRD-P0-002 更新为 `已缓解`；若只完成代码但缺运行验证，保持 `待验证`。 |

阶段 2 的实施计划需要覆盖：

- 幂等键生成和保存位置。
- regenerate 与 create 的去重边界。
- worker 崩溃、超时、重启后的任务恢复规则。

## 阶段 3：复习数据一致性

| 项目 | 内容 |
| --- | --- |
| 目标 | 解决 attempt、feedback 和 reviewSession 并发写入可能丢数据的问题。 |
| 触及风险 ID | PRD-P0-003 |
| 主要改动 | 选择并实现一种一致性策略：章节 JSON 乐观锁、事务内行锁，或拆分 attempt/feedback 独立表；补并发提交测试。 |
| 验收标准 | 同一 review session 并发提交 attempts 不会丢记录；feedback 与 attempt 并发时不会覆盖对方；冲突响应可被客户端理解和重试。 |
| 审计台账更新 | PRD-P0-003 只有在并发测试通过后才能更新为 `已缓解`。 |

阶段 3 的实施计划需要先做取舍记录：

- 如果选择乐观锁，必须定义版本字段、冲突状态码和客户端重试语义。
- 如果选择行锁，必须定义事务范围和超时策略。
- 如果选择拆表，必须定义与现有 `chapter_json` 的兼容和迁移策略。

## 阶段 4：安全流量治理

| 项目 | 内容 |
| --- | --- |
| 目标 | 降低滥用、超大请求、跨域误用和匿名设备伪造带来的成本与数据边界风险。 |
| 触及风险 ID | PRD-P0-004 |
| 主要改动 | 增加请求体大小限制、基础 rate limit、CORS 收紧策略、deviceId 格式约束和高风险接口保护。 |
| 验收标准 | 超大 body 会返回明确错误；短时间重复提交会被限流；生产 CORS 不再默认放开所有来源；删除数据接口有更明确的保护边界。 |
| 审计台账更新 | PRD-P0-004 根据安全测试结果更新为 `待验证` 或 `已缓解`，并记录仍接受的匿名设备风险。 |

阶段 4 的实施计划需要明确：

- 本地开发、HTML Demo、iOS 真机、生产域名的 CORS 差异。
- rate limit 的粒度：deviceId、IP、接口、生成任务类型。
- 哪些风险留到账号系统后解决。

阶段 4 最小实现记录：

- 请求体读取统一走带上限的 JSON reader，默认普通 API 1MB，生成 API 2MB，可通过 `SHIBEI_MAX_JSON_BODY_BYTES` 与 `SHIBEI_MAX_GENERATION_BODY_BYTES` 覆盖。
- 增加单实例内存 rate limit，当前粒度为 `routeGroup + deviceId + clientIp`。默认生成接口为 10 分钟 12 次，推荐文章导入为 10 分钟 30 次，普通 mutation 为每分钟 120 次，普通 API 为每分钟 300 次；可通过对应环境变量调整。
- 高风险 API 要求 deviceId 使用安全字符集和 128 字符以内长度；健康检查、静态资源、推荐文章公开读取不强制 deviceId。
- CORS 在非生产环境继续按请求 Origin 返回，生产环境改为读取 `SHIBEI_ALLOWED_ORIGINS` / `SHIBEI_ALLOWED_ORIGIN` / `SHIBEI_PUBLIC_BASE_URL`。
- 当前 rate limit 适用于 Railway 单副本；如果后续扩展到多副本，需要迁移为 Postgres 或 Redis 级别的分布式限流。
- 仍需在真实 production 环境验证：iOS 真机请求、推荐文章封面、GitHub/Railway smoke、HTML demo 是否都处于预期 CORS 策略下。

## 阶段 5：观测与压测

| 项目 | 内容 |
| --- | --- |
| 目标 | 建立能证明生产状态的最小观测和压测证据，而不是只靠代码推断。 |
| 触及风险 ID | PRD-P0-001、PRD-P0-002、PRD-P0-003、PRD-P0-004、PRD-P0-005 |
| 主要改动 | 增加结构化日志、请求相关 ID、生成链路分段耗时、队列指标、模型错误分类、基础压测脚本和压测报告模板。 |
| 验收标准 | 能看到提交接口延迟、队列等待时间、生成耗时、模型错误率、worker 成功/失败数、DB 错误和 APNS 发送结果；至少完成 10/50 并发压测记录。 |
| 审计台账更新 | PRD-P0-005 必须有指标和告警证据后才能更新为 `已缓解`；其他 P0 风险根据压测结果补充运行证据。 |

阶段 5 的实施计划需要定义：

- 最小 SLI：提交接口 p95、生成完成 p95、任务失败率、队列等待时间、模型错误率。
- 日志字段：requestId、deviceId 后缀或 hash、chapterId、jobId、stage、durationMs、errorCode。
- 压测报告保存位置和复查节奏。

## 阶段完成定义

一个阶段只有同时满足以下条件，才算完成：

1. 代码或文档改动已经合并到主工作流。
2. 阶段验收标准全部通过。
3. 对应测试、压测或人工验证结果已经记录。
4. `docs/production-readiness-review-zh.md` 已更新风险状态和证据。
5. 若引入新运行命令、环境变量或部署步骤，README 或 Railway 文档已同步。

## 暂不纳入 P0 的事项

- 正式账号系统和跨设备数据迁移。
- Redis/BullMQ 或云托管队列选型。
- 完整数据仓库、BI 或长期成本分析平台。
- App Store 正式审核材料之外的对外生产成熟度说明。
- 大规模多区域部署、高可用数据库和灾备自动化。
