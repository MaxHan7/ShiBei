# 拾贝生产就绪度审计台账

> 这份文档用于持续记录拾贝从 Railway 云端原型走向生产级后端的风险、证据、验证计划和整改状态。它不是对外宣传说明，也不是完整工程实施方案；所有状态只反映当前仓库可见证据。

## 当前结论

当前后端适合小流量真机验证和受控内测，不适合直接承诺正式生产环境的高并发、低延迟或高可用 SLA。

已经具备的生产意识：

- iOS 主入口 `POST /api/chapters` 会先返回 `submitted`，后台继续生成，避免客户端一直阻塞等待完整模型链路。
- Railway 原型支持 PostgreSQL 按匿名设备 ID 持久化章节、通知、生成任务和复习状态。
- 生成任务有阶段状态、模型请求超时和整次生成超时。
- 服务启动时会把上次重启前未完成的生成任务标记为中断，避免章节永久停在处理中。

仍缺少的生产证据：

- 没有真实流量模型、SLO、容量规划或压测报告。
- 没有正式异步任务队列、背压、租户配额或分布式任务协调。
- 没有完整观测闭环，包括指标、结构化日志、trace、告警和生成耗时分段。
- 没有安全、数据治理、故障注入和客户端弱网恢复专项审计。

## 审计标准

当前排查只达到架构初筛：已经能识别主要生产隐患，但不能证明系统能扛住生产流量。阶段 0 接口和状态基线记录见 `docs/production-hardening-stage-0-baseline-zh.md`。后续每个风险项需要补齐三类证据：

1. 代码证据：对应代码、配置或文档边界清楚。
2. 测试证据：有自动化测试、压测或故障注入结果。
3. 运行证据：有真实或模拟环境的指标、日志和告警记录。

风险状态说明：

| 状态 | 含义 |
| --- | --- |
| 已识别 | 已从代码或文档中确认存在风险。 |
| 待验证 | 风险合理但还缺少压测、故障注入或运行数据。 |
| 已缓解 | 已有代码、测试和运行证据证明风险被控制。 |
| 接受风险 | 明确接受该风险，并记录边界、原因和复查时间。 |

## 验证计划

### 并发压测

- 场景：10、50、100 个设备同时提交 `POST /api/chapters`，每个设备使用不同 `X-Device-Id`。
- 输入：短文本、长文本、文章链接、公众号链接分组测试。
- 记录：请求成功率、提交接口 p95、生成完成 p95/p99、失败原因、模型错误率、DB 连接、内存和 CPU。
- 合格线：先由产品和工程定义内测目标；没有目标前不得宣称可承载高并发。

### 重复请求和幂等

- 场景：同一设备对同一正文快速重复提交、同一章节重复触发 regenerate、客户端超时后重试。
- 记录：是否生成重复章节、是否出现状态覆盖、是否重复扣模型成本。
- 目标：明确哪些请求允许重复，哪些请求需要幂等键或服务端去重。

### 服务重启

- 场景：生成任务进行到 `generating_questions`、`quality_checking`、`auto_regenerating_questions` 时重启服务。
- 记录：章节是否进入明确失败态，用户是否能重新生成，未完成 `generation_jobs` 是否被标记中断。
- 目标：不能永久停留在处理中，也不能悄悄丢失用户章节。

### DB 短暂不可用

- 场景：Postgres 连接失败、慢查询、连接池耗尽。
- 记录：创建章节、轮询章节、提交答题、删除数据的错误文案和恢复行为。
- 目标：用户看到明确错误；服务不进入未知半写入状态。

### 模型超时和 provider 错误

- 场景：模型请求超时、429、5xx、返回非 JSON、返回结构不符合 schema。
- 记录：章节最终状态、failureReason、后台错误分类、是否有重试和成本记录。
- 目标：失败可解释、可重试、可观测，不把 provider 错误混成同一类。

### 客户端轮询恢复

- 场景：提交生成后切后台、断网、杀 App、重进 App、推送失败。
- 记录：章节列表和详情能否恢复最新状态，是否需要手动刷新，是否出现重复提交。
- 目标：iOS 以服务端章节状态为准，前台恢复时能重新同步。

## 整改路线图

P0 生产化的分阶段实施顺序见 `docs/production-hardening-plan-zh.md`。本节保留风险整改方向；每完成一个阶段，都需要回到本台账更新对应风险状态和证据。

### P0：受控内测前

- 为生成任务增加正式队列、worker、并发上限和任务领取机制。
- 为创建章节和重新生成加入幂等策略，避免重复提交导致重复生成。
- 为复习 attempt、feedback 或章节 JSON 写入增加版本控制、事务锁或独立表设计。
- 增加请求体大小限制、基础 rate limit、deviceId 防滥用策略和更严格 CORS 配置。
- 建立最低可用观测：结构化日志、生成耗时分段、错误分类、模型调用统计、告警入口。

### P1：公开 beta 前

- 增加模型 provider 429/5xx/超时的重试退避、熔断和错误分类。
- 限制 Playwright 抓取并发，评估外部文章提取服务或异步抓取池。
- 完善 APNS 交付指标、失败聚合和用户侧降级提醒。
- 补齐 iOS 弱网、前后台、重复提交和轮询退避策略。
- 完成隐私、日志脱敏、删除证明和数据保留周期审计。

### P2：正式上线前

- 形成容量模型和压测报告，明确内测、beta、正式发布的流量上限。
- 建立备份恢复、迁移演练、事故响应和回滚流程。
- 将生产配置、环境变量、数据库迁移和监控告警纳入发布 checklist。
- 对安全、数据治理、Swift 客户端可靠性和后端架构做专项复审。

## 后续维护规则

- 每次生产相关代码或部署策略变更后，更新对应风险项状态和证据。
- 风险降级必须同时具备代码变更、测试结果和运行证据。
- 新增风险使用 `PRD-P0-xxx`、`PRD-P1-xxx` 或 `PRD-P2-xxx` 编号，避免覆盖历史编号。
- 不用“已经支持生产”这类模糊表述，除非有压测、故障注入和运行监控数据支撑。
